## 개발 노트 작성 원칙

개발 노트(data/content/notes/*.md)를 작성할 때 아래 원칙을 따른다.

### 톤
- 존댓말 기조 + 자연스러운 구어체 ("생각해보면", "어찌보면", "~했거든요")
- 설명문/보고서 톤이 아닌 **개발 일지/이야기 톤**
- 독자에게 직접 말을 거는 표현은 지양 ("여러분", "~해보세요" 등 불필요)

### 구조
- **"문제 → 해결 → 새 문제" 반복 패턴** — 실제 개발 여정을 따라가는 흐름
- 섹션 제목은 행동/상황 중심 ("NestJS에서 시작했다가 Fastify로")
- 섹션 간 전환은 자연스럽게 — 이전 섹션의 결과가 다음 섹션의 문제로 이어짐

### 문장
- 짧은 문장과 긴 설명의 **리듬감** — 핵심 결론은 짧게, 배경 설명은 길게
- 기술 용어는 괄호로 영문 원명 병기: "ESM(ECMAScript Modules)"
- **굵은 글씨**로 핵심 포인트 강조
- 수치가 있으면 구체적으로 표현: "286개 → 144개 → 27개"

### 코드/다이어그램
- 코드는 **핵심 부분만 발췌**, 전체 구현 노출 금지 (회사 자산 보호)
- Mermaid 다이어그램은 아키텍처/흐름을 한눈에 파악할 수 있을 때만 사용
- 데이터 구조나 설정은 테이블로 정리

### 금지 사항
- 회사 코드/설정의 세부 스펙 노출 금지
- "~합니다" 체의 딱딱한 보고서 톤 금지
- 근거 없는 비교 ("X보다 Y가 좋다" — 직접 경험하지 않은 비교는 하지 않음)
