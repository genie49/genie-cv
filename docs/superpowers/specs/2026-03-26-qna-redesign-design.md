# QnA 섹션 리디자인 스펙

## 배경

현재 QnA 섹션은 11개의 범용적 질문으로 구성되어 있으나, 다음 문제점이 있다:

- 정량적 임팩트 없이 정성적 서술에 머묾
- "실행 > 완벽" 관점이 약하고, "구조 먼저" 메시지가 지나치게 강함
- 멀티에이전트 관련 질문이 3개로 중복
- 팀 협업/피드백 문화 경험이 부족
- 한 답변에 3~4개 프로젝트를 나열하여 산만

## 목표

- 범용 포트폴리오를 유지하면서 토스가 중시하는 요소(임팩트, 빠른 실행, 근원적 질문, 피드백 문화)를 자연스럽게 녹여냄
- 11개 → 7개로 축소, 각 답변을 하나의 에피소드(STAR 구조)로 집중
- Self Q&A 자문자답 톤 유지

## 변경 범위

- `data/qna.json` — 질문/답변 데이터 전면 교체
- `apps/client/src/pages/QnAPage.tsx` — 변경 없음 (데이터만 바뀜)
- `scripts/embed.ts` 재실행 — RAG 임베딩 갱신 필요

## 질문 구조 (7개)

| # | 질문 | 메인 에피소드 | 간접 매핑 토스 가치 |
|---|---|---|---|
| 1 | 간단한 자기소개를 해주세요. | 전체 커리어 — 각 프로젝트에서 해결한 문제를 한 마디로 압축 | 미션 지향 |
| 2 | 개발할 때 어떤 스타일로 일하나요? | kimpro 이벤트 드리븐 결정 — 뼈대 먼저, 불확실한 건 만들면서 검증 | 실행 > 완벽 |
| 3 | 가장 도전적이었던 프로젝트는? | fingoo — 금융 데이터 + 에이전트 예측 불가능성, 전체 파이프라인 완성 | 기대를 뛰어넘는 수준 |
| 4 | 가장 큰 실패 경험과 거기서 배운 점은? | fingoo 만능 에이전트 실패 → Supervisor 패턴 재설계 | 주도적 학습 + 근원적 질문 |

| 6 | 기술 스택을 선택할 때 어떤 기준으로 결정하나요? | bonda Qdrant vs 포트폴리오 LanceDB — 같은 벡터 검색, 다른 최적해 | 임팩트 집중 |
| 7 | AI를 활용해 실제로 해결한 비즈니스 문제는? | kimpro 마케팅 4단계 수동 프로세스 → 에이전트 자동화 | 임팩트 집중 |

### 삭제된 질문과 이유

| 기존 질문 | 삭제 이유 |
|---|---|
| 본인의 강점은 무엇인가요? | 자기소개와 중복. 강점은 에피소드 속에서 자연스럽게 드러남 |
| 앞으로의 목표는 무엇인가요? | 범용적이고 토스 핏과 거리가 있음 |
| 멀티에이전트 시스템을 설계할 때 가장 중요하게 고려하는 점은? | 실패 경험(Q4) 답변에 통합 |
| RAG 파이프라인 설계 시 가장 어려웠던 점은? | 기술 선택(Q6) 답변에 일부 통합 |
| 레거시 시스템을 리빌드한 경험을 설명해주세요. | 자기소개에서 간접 언급으로 충분 |

## 답변 포맷

### STAR 기반 에피소드 구조

```
[상황] 어떤 맥락이었는지 1~2문장
[판단] 어떤 선택지가 있었고 왜 그 길을 골랐는지
[행동] 구체적으로 무엇을 했는지
[결과/배움] 어떤 변화가 생겼고 무엇을 얻었는지
```

STAR 라벨은 명시하지 않고 자연스러운 서술체로 녹인다.

### 톤 가이드라인

| 항목 | 현재 | 변경 |
|---|---|---|
| 길이 | 평균 200자+ 장문 | 120~180자, 에피소드 하나에 집중 |
| 프로젝트 언급 | 한 답변에 3~4개 나열 | 1개 메인 + 필요시 1개 보조 |
| 기술 용어 | 나열형 | 판단 근거 중심 |
| 마무리 | 교훈/원칙 장문 서술 | 짧고 임팩트 있는 한 문장 |
| 첫 문장 | 경험 시작부터 서술 | 핵심 메시지를 첫 문장에 배치 |

## 각 답변 상세 방향

### Q1. 간단한 자기소개를 해주세요.

- **핵심 메시지**: "현장의 병목을 AI로 해결하는 개발자"
- 각 프로젝트를 해결한 문제 중심으로 한 줄씩 압축
- 마무리: "기술보다 문제를 먼저 봅니다"

### Q2. 개발할 때 어떤 스타일로 일하나요?

- **핵심 메시지**: "설계의 뼈대를 빠르게 잡고, 불확실한 부분은 만들면서 검증한다"
- **메인 에피소드**: kimpro 이벤트 드리븐 구조 결정 — 직접 호출 vs Pub/Sub 논의에서 뼈대를 먼저 잡고, 세부 토픽 설계는 서비스를 붙이면서 확정
- 불확실한 부분을 별도로 기록해두고 개발하면서 해소하는 습관 언급
- 마무리: "완벽한 설계를 기다리면 실행이 늦고, 설계 없이 뛰면 되돌아온다. 그 사이를 찾는 게 제 스타일입니다"

### Q3. 가장 도전적이었던 프로젝트는?

- **핵심 메시지**: "금융 데이터의 복잡성 + 에이전트의 예측 불가능성을 동시에 다뤄야 했다"
- **메인 에피소드**: fingoo — 이질적 데이터 소스 통합, 에이전트 간 간섭, 프론트~인프라 전체 파이프라인 완성
- 마무리: "끝까지 만들어봐야 보이는 것들이 있다"

### Q4. 가장 큰 실패 경험과 거기서 배운 점은?

- **핵심 메시지**: "만능 에이전트라는 가정이 틀렸다"
- **메인 에피소드**: fingoo 만능 에이전트 → 프롬프트 튜닝 2주 악순환 → Supervisor 패턴 재설계
- 보조: 이후 kimpro에서 처음부터 이 구조를 채택
- 마무리: "그 실패가 지금의 설계 원칙이 되었다"

### Q5. 팀에서 어떻게 협업하나요?

- **핵심 메시지**: "솔직한 피드백을 주고받는 환경에서 가장 잘 일한다"
- **에피소드 1**: fingoo — PR 코드리뷰, 디스코드에서 가감없이 피드백, 흥미로운 레포/소식 공유하며 아이디어 교환
- **에피소드 2**: 아이머스 — 인턴이었지만 회의를 직접 주체하는 권한 활용, 설계 내용을 세미나로 공유
- 기존 "의견 충돌 해결"(WebSocket vs SSE) 에피소드는 삭제
- 마무리: "직급이나 경력보다 근거 있는 의견이 존중받는 팀에서 가장 좋은 결과가 나왔다"

### Q6. 기술 스택을 선택할 때 어떤 기준으로 결정하나요?

- **핵심 메시지**: "유행이 아니라 문제에서 출발한다"
- **메인 에피소드**: bonda에서 듀얼 벡터 검색이 필요해 Qdrant 선택 vs 포트폴리오 사이트에서 운영 비용 제로를 위해 LanceDB 선택 — 같은 벡터 검색이라도 요구사항에 따라 답이 다름
- 마무리: "기술 선택은 트레이드오프를 이해하고 현재 상황의 최적해를 찾는 과정"

### Q7. AI를 활용해 실제로 해결한 비즈니스 문제는?

- **핵심 메시지**: "AI 자동화의 출발점은 기술이 아니라 현장 관찰"
- **메인 에피소드**: kimpro — 마케팅팀의 4단계 수동 프로세스 관찰 → 각 단계에서 사람의 판단을 에이전트 노드로 변환 → 스냅샷/롤백 구조
- 마무리: "어디서 시간을 쓰고 있는가를 관찰하는 것이 출발점"
