# AI에게 500페이지를 주지 마, 필요한 2페이지만 줘

같은 모델이라도 컨텍스트를 어떻게 구성하느냐에 따라 결과가 완전히 달라진다. Andrej Karpathy가 말했듯, LLM이 CPU라면 컨텍스트 윈도우는 RAM이다. RAM에 무엇을 올리느냐가 CPU의 성능을 결정한다.

프롬프트 엔지니어링이 "모델에게 뭘 말할까"였다면, **컨텍스트 엔지니어링은 "모델이 판단할 때 무엇을 보고 있게 할까"다.** Aimers의 멀티에이전트 시스템에서 컨텍스트를 설계한 과정을 정리한다.

## 동적 시스템 프롬프트 — 캐싱을 깨지 않으면서 실시간 정보를 주입하는 법

에이전트의 시스템 프롬프트에는 도메인 지식, 도구 사용법, 서브에이전트 가이드가 포함되어 수천 토큰에 달한다. 여기에 현재 시간, 캠페인 상태 같은 실시간 정보도 필요하다.

그런데 시스템 프롬프트가 매번 바뀌면 **Anthropic Prompt Caching이 무효화**된다. 캐시는 프롬프트의 해시가 동일해야 히트가 발생하기 때문이다. 캐싱이 깨지면 매 요청마다 수천 토큰의 입력 비용을 전액 지불해야 한다.

해결은 **불변과 가변을 물리적으로 분리**하는 것이었다. 시스템 프롬프트의 불변 부분(도메인 지식, 도구 가이드)은 캐시 태그를 달아 고정하고, 동적 정보(현재 시간, 캠페인 상태)는 미들웨어가 별도 블록으로 뒤에 추가한다. 캐시 키는 태그된 블록까지만 사용하므로, 뒤에 붙는 동적 부분은 캐시를 깨지 않는다.

이 구조 하나로 **시스템 프롬프트 캐시 히트율이 90% 이상**을 유지하면서, 에이전트는 항상 최신 컨텍스트를 가진 상태로 판단한다. 한 세션에서 15번 호출하면 14번은 캐시 히트가 발생해, 입력 토큰 비용이 약 80~90% 절감된다.

## 도구 결과는 또 다른 지시다

에이전트가 외부 API를 호출하면 응답이 돌아온다. 이 응답을 그대로 컨텍스트에 넣으면 어떻게 될까. 크롤링 결과 수천 토큰, 검색 결과 수천 토큰 — 몇 번만 도구를 호출해도 컨텍스트가 폭발한다.

**컨텍스트 윈도우는 유한한 자원**이다. 도구 결과를 전부 넣으면 에이전트의 판단력이 흐려진다. 정보가 많아지면 모델은 "뭘 봐야 하지?"가 되고, 정작 중요한 맥락을 놓친다.

그래서 도구 설계를 바꿨다. **상태를 먼저 저장하고, 에이전트에게는 핵심 결과만 반환**한다. 외부 API 응답의 전체 데이터는 스냅샷에 저장해두고, 에이전트에게는 "저장 완료. 다음 단계로 넘어가면 된다" 수준의 요약만 돌려준다. 자세한 내용이 필요하면 조회 도구로 다시 가져올 수 있다.

도구 결과를 이렇게 설계하면 **도구의 반환값 자체가 에이전트에게 주는 지시**가 된다. "이 데이터를 분석해"가 아니라 "저장됐으니 다음 단계를 진행해"라고 말해주는 것이다. 에이전트가 어디로 가야 할지를 도구 결과가 안내한다.

## 에러도 컨텍스트다

LangChain의 기본 동작은 도구 에러가 발생하면 에러 메시지를 ToolMessage로 반환한다. 그런데 여기에 **성공/실패 구분이 없다.** LLM 입장에서는 도구 응답이 정상인지 에러인지 알 수 없어, 에러 메시지를 정상 결과로 오해하고 사용자에게 전달하는 일이 생겼다.

미들웨어로 모든 도구 실행을 감싸서, 성공 시 `status: success`를, 실패 시 `status: error`와 구체적 에러 메시지를 넣었다. 이제 LLM은 에러를 인지하고 재시도하거나 사용자에게 상황을 안내할 수 있다.

에러 메시지의 구체성도 중요하다. "잘못된 값입니다"라는 모호한 에러는 에이전트를 멈추게 하지만, "이 필드는 숫자여야 하는데 문자열을 받았습니다"라는 구체적 에러는 에이전트가 스스로 수정할 수 있는 **결정적 힌트**가 된다.

## interrupt 후 컨텍스트 정합성

Human-in-the-Loop 패턴에서 워크플로우가 interrupt로 중단되었다가 재개되면, 체크포인트에 "도구를 호출했지만 결과가 없는" 메시지가 남는다. Anthropic API는 모든 tool_use에 대응하는 tool_result가 있어야 한다는 엄격한 규칙이 있어, 이 상태로는 API 호출이 실패한다.

미들웨어가 모델 호출 직전에 메시지를 순회하면서, 결과가 없는 orphaned tool_use를 찾아 합성 응답을 주입한다. "이 도구 호출은 interrupt로 인해 실행되지 않았습니다"라는 메시지를 넣으면, LLM이 상황을 이해하고 자연스럽게 다음 단계를 진행한다.

이 미들웨어 없이는 **interrupt 후 재개가 불가능**하다. Human-in-the-Loop의 기반이 되는 컨텍스트 정합성 장치다.

## 돌이켜보면

컨텍스트 엔지니어링의 핵심은 결국 **"적은 토큰으로 더 나은 판단"**이다.

- 시스템 프롬프트를 불변/가변으로 분리해 **캐싱 효율과 실시간성을 동시에** 확보
- 도구 결과를 요약해서 반환해 **컨텍스트를 날씬하게** 유지
- 에러에 status를 부여해 **LLM이 상황을 정확히 인지**
- interrupt 후 깨진 메시지를 패치해 **컨텍스트 정합성을 보장**

프롬프트를 잘 쓰는 것도 중요하지만, **모델이 보는 전체 그림을 설계하는 것**이 에이전트 품질을 결정한다.
