# AI를 시스템에 맞추지 말고, 시스템을 AI에게 맞춰라

처음에는 기존 시스템에 AI를 끼워 넣으려 했다. Chat 서비스 안에서 에이전트를 실행하고, LangGraph의 state에 캠페인 데이터를 담고, 프레임워크가 제공하는 방식대로 동작하게 만들었다. 그런데 AI는 기존 시스템의 가정과 맞지 않았다. **시스템에 AI를 맞추니 성능도 UX도 나빠졌다.** 결국 시스템을 AI에 맞춰 다시 설계하게 된 과정을 정리한다.

## Chat이 전부 하던 시절

처음 구조는 단순했다. Chat 서비스가 WebSocket으로 사용자와 연결되고, 같은 프로세스에서 AI 에이전트를 실행하고, 결과를 사용자에게 전달했다. 하나의 서비스가 실시간 상태 전달과 AI 실행을 모두 담당했다.

문제는 **두 역할의 관심사가 완전히 다르다**는 점이었다. Chat은 사용자에게 지금 무슨 일이 일어나고 있는지를 실시간으로 전달하는 게 핵심이다. 가벼워야 하고, 연결이 안정적이어야 한다. 반면 Workflow는 LLM을 돌리고, 도구를 호출하고, 서브에이전트를 관리하는 무거운 작업이다. 미들웨어, 스냅샷, 에러 복구 같은 복잡한 인프라가 필요하다.

둘이 한 프로세스에 묶여 있으니 **양쪽 다 발전시키기 어려웠다.** Workflow 쪽에 미들웨어를 추가하면 Chat의 연결 안정성에 영향을 줄까 걱정해야 했고, Chat을 스케일링하면 Workflow의 LLM 부하까지 따라 올라갔다. 종속성이 높아질수록 에이전트 인프라(harness)를 자유롭게 설계하기 어려워졌다.

## 관심사를 분리하다

MSA의 기본 원칙대로 관심사를 나눴다. Chat은 **실시간 전달**에 집중하고, Workflow는 **에이전트 실행**에 집중한다. 분리하니까 Workflow 쪽에서 미들웨어 파이프라인, 스냅샷 시스템, 서브에이전트 오케스트레이션 같은 harness를 자유롭게 설계할 수 있게 됐다. Chat의 연결 안정성을 걱정하지 않아도 됐다.

그런데 분리된 둘을 어떻게 이을 것인가.

## 분리된 둘 사이에 메시지 큐를 놓다

HTTP 호출은 옵션이 아니었다. LLM은 한 번의 응답을 생성하는 데 **수십 초에서 수 분**이 걸린다. 그 동안 HTTP 커넥션을 붙잡고 있어야 하고, Workflow 워커가 재시작되면 커넥션이 끊기면서 진행 중인 응답이 유실된다.

**GCP Pub/Sub**을 사이에 놓았다. Workflow가 메시지를 발행하면 Chat이 구독해서 사용자에게 전달한다. 워커가 재시작되어도 Chat의 WebSocket 연결은 유지되고, 메시지는 버퍼링된다.

여기에 다대다 통신 요구도 있었다. 같은 캠페인을 여러 사용자가 동시에 보는 경우, 하나의 Workflow 응답이 **여러 Chat 인스턴스에 동시에 전달**되어야 한다. Pub/Sub의 속성 기반 필터링으로, 단일 토픽에서 인스턴스별 라우팅을 해결했다.

독립 스케일링은 분리의 부산물이었다. WebSocket 연결 수가 늘면 Chat만 스케일아웃하고, LLM 부하가 늘면 Workflow 워커만 추가하면 된다.

## 인스턴스가 여러 개면, 누가 어디 있는지 어떻게 아는가

Chat 인스턴스 3개, Workflow 인스턴스 2개가 떠 있다고 하자. 사용자 A가 Chat-2에 연결되어 있고, 그 요청을 Workflow-1이 처리하고 있다. Workflow-1이 결과를 발행하면 **Chat-2가** 받아야 한다. Chat-1이나 Chat-3이 받으면 안 된다 — 사용자 A의 WebSocket 연결은 Chat-2에만 있으니까.

Pub/Sub의 속성 필터링만으로는 부족했다. 어떤 Chat 인스턴스가 어떤 사용자를 담당하고 있는지를 **동적으로 알아야** 한다. 인스턴스는 스케일링에 따라 생겼다 사라지니까.

**Redis 기반 서비스 디스커버리**로 해결했다. 각 Chat 인스턴스는 시작 시 자신의 인스턴스 ID를 Redis에 등록하고, 주기적으로 heartbeat를 갱신한다. 사용자가 WebSocket으로 연결되면 `user:{userId} → chat-instance-2` 매핑이 Redis에 기록된다. Workflow가 결과를 발행할 때 이 매핑을 조회해서, 해당 Chat 인스턴스가 구독하는 Pub/Sub 채널로 정확히 라우팅한다.

같은 캠페인을 여러 사용자가 동시에 보는 경우도 있다. 이때는 캠페인 ID로 연결된 모든 사용자를 조회하고, 각 사용자가 연결된 Chat 인스턴스로 **팬아웃**한다. 인스턴스가 죽으면 heartbeat가 만료되면서 자동으로 레지스트리에서 제거된다. 새 인스턴스가 뜨면 등록만 하면 된다 — 기존 인스턴스의 설정을 건드릴 필요가 없다.

수평 확장의 핵심은 **인스턴스 간 직접 통신이 없는 것**이었다. 모든 라우팅 정보가 Redis에 있고, 메시지는 Pub/Sub을 거친다. Chat을 3개에서 10개로 늘려도 Workflow 코드는 한 줄도 바뀌지 않는다.

## AI는 병렬로 일한다, 그런데 state가 허락하지 않았다

서비스 분리는 끝났다. 다음 문제는 Workflow 내부에서 터졌다.

에이전트가 도구를 호출할 때, Supervisor가 "제품도 분석하고 전략도 세워"라고 판단하면 두 서브에이전트가 **동시에** 실행된다. 각각이 캠페인 데이터를 수정한다. 문제는 LangGraph의 state 관리 방식이었다.

**LangGraph state는 replace 방식**이다. 캠페인 데이터 전체를 하나의 state로 담고, 도구가 결과를 반환하면 통째로 교체한다. 두 도구가 병렬로 실행되면, 먼저 끝난 쪽의 변경이 나중에 끝난 쪽에 의해 **덮어써진다.** 데이터 유실.

첫 번째 시도는 **AI를 시스템에 맞추는 것**이었다. 도구가 직접 state를 수정하는 대신, 변경사항을 배열로 만들어서 반환하게 했다. 이렇게 하면 state 충돌은 피할 수 있지만, **병렬 도구 호출을 포기해야 했다.** 순차 실행.

결과는 처참했다. **속도가 약 2.5배 느려졌다.** 서브에이전트 5개가 동시에 3~5개 도구를 호출하던 것을 하나씩 순서대로 실행하니 당연한 결과였다. 성능만 나빠진 게 아니었다. UX도 달라졌다. 병렬이면 "제품 분석 시작", "전략 수립 시작"이 거의 동시에 뜨면서 AI가 활발하게 일하는 느낌이었는데, 순차로 바꾸니 **한참 조용하다가 갑자기 결과가 툭 나타나는** 느낌이었다.

2.5배의 속도와 실시간 피드백의 생동감. 이걸 포기할 수 없었다.

## 시스템을 AI에게 맞추다

발상을 바꿨다. LangGraph state에 맞추지 말고, **state를 대체하자.**

워크플로우가 시작되면 캠페인 데이터를 **Redis에 스냅샷**으로 복사한다. 이후 모든 변경은 Redis에만 기록한다. 핵심은 **Changes 리스트**다. 도구가 데이터를 수정할 때 전체를 교체하는 게 아니라, `product.name = "새 값"` 같은 필드 단위 변경을 리스트에 순서대로 쌓는다.

| LangGraph state (Before) | Redis 스냅샷 + Changes (After) |
|---|---|
| 전체 교체 (replace) | 필드 단위 추가 (append) |
| 병렬 쓰기 시 덮어씀 | 병렬 쓰기가 리스트에 순서대로 쌓임 |
| 병렬 도구 호출 불가 | 병렬 도구 호출 가능 |

읽기할 때는 원본 스냅샷에 Changes를 순서대로 적용한 결과를 반환한다. 서브에이전트 A가 `product.name`을 수정한 후 서브에이전트 B가 같은 필드를 읽으면 최신 값을 볼 수 있다.

같은 필드를 두 에이전트가 동시에 수정하면? 이론적으로는 나중에 쓴 값이 이기지만, 실제로는 일어나지 않는다. 각 서브에이전트의 도구에 **도메인 제한**이 걸려 있기 때문이다. 제품 분석가는 `product.*`만, 전략가는 `brief.*`만 수정할 수 있다. 도구 레벨에서 도메인을 격리하면 물리적으로 같은 필드를 건드리는 일이 발생하지 않는다.

워크플로우가 정상 완료되면 스냅샷 + Changes를 합산해서 DB에 한 번에 커밋한다. 실패하면 Redis만 정리하면 된다 — DB는 아직 건드리지 않았으므로 원본이 안전하다.

## 돌이켜보면

두 번의 설계 전환이 있었다. Chat과 Workflow를 분리한 것, 그리고 LangGraph state를 Redis로 대체한 것. 둘 다 같은 깨달음에서 나왔다. **AI를 기존 시스템의 제약에 맞추면, AI가 할 수 있는 것의 상당 부분을 포기하게 된다.**

기존 HTTP 통신에 맞추면 AI의 긴 처리 시간과 실시간 스트리밍을 포기해야 했다. LangGraph state에 맞추면 병렬 실행과 실시간 피드백을 포기해야 했다. 시스템을 AI에 맞춰 다시 설계하니, 포기했던 것들이 돌아왔다.
