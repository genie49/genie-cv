# 음성 한마디에 14가지 행동 — Koog Agent와 Tool Calling

"내일 오후 3시에 회의 일정 잡아줘." 이 한마디가 실제로 Google Calendar에 일정을 생성하려면, LLM이 의도를 파악하고, 적절한 도구를 선택하고, 인자를 채워서 실행해야 한다. Koog 프레임워크로 14개 도구를 등록하고, Gemini Function Calling으로 이 과정을 자동화한 설계를 정리한다.

## 서버 없이 앱 안에서 돌린다

Koog를 선택한 이유는 단순하다. **서버를 하나 더 두고 싶지 않았다.** LangChain은 Python 서버가 필요하고, 도구가 전화를 걸거나 앱을 조작하려면 서버에서 다시 앱으로 통신해야 한다. Koog는 Kotlin 네이티브라 앱 안에서 바로 실행된다. 도구의 실행 함수에서 안드로이드 API를 직접 호출할 수 있다.

## 14개 도구의 영역

| 영역 | 도구 | 확인 필요 |
|---|---|---|
| 연락처 | 검색 | - |
| 전화/메시지 | 전화 걸기, SMS, 카카오톡 | 전부 필요 |
| 캘린더 | 일정 조회/생성/수정/삭제 | - |
| 태스크 | 할일 조회/생성/완료/삭제 | - |
| 알림 | 디바이스 알림 읽기 | - |
| 앱 제어 | 범용 앱 조작 | - |

14개 도구가 5개 영역을 커버한다. 에이전트가 질문을 분석해서 어떤 도구를 어떤 순서로 호출할지 판단한다. "엄마한테 전화해줘"라면 연락처 검색 → 전화 걸기가 순차로 실행된다.

## 도구 간 의존 관계가 있다

단순히 도구를 나열하면 에이전트가 잘못된 순서로 호출할 수 있다. "엄마한테 전화해"에서 연락처를 검색하지 않고 바로 전화를 걸려고 하면 전화번호가 없다.

시스템 프롬프트에 **도구 의존 규칙**을 명시한다. "전화를 걸기 전에 반드시 연락처를 먼저 검색해라", "일정을 수정하기 전에 먼저 조회해라" — 이런 규칙이 에이전트의 도구 호출 순서를 안내한다.

## 대화 히스토리를 슬라이딩 윈도우로 관리한다

음성 비서는 연속 대화가 가능해야 한다. "내일 회의 잡아줘" → "시간은 3시로" → "장소도 추가해줘" — 이전 맥락을 기억해야 한다.

대화 히스토리를 **최근 10턴 슬라이딩 윈도우**로 유지한다. 모든 히스토리를 보관하면 토큰 비용이 늘어나고 컨텍스트 윈도우를 초과할 수 있다. 10턴이면 대부분의 연속 대화를 커버하면서도 비용을 통제할 수 있는 균형점이다.

히스토리는 Room DB에 저장해서 앱을 재시작해도 유지된다.

## 싱글톤 도구에 Context를 넣는 딜레마

Koog의 도구는 싱글톤(object)으로 정의해야 한다. 그런데 전화를 걸거나 앱을 실행하려면 안드로이드 Context가 필요하다. 싱글톤에 Context를 직접 저장하면 메모리 누수 위험이 있다.

Application Context를 사용해서 해결했다. Application은 앱 전체 생명주기와 동일하므로 Activity 누수가 발생하지 않는다. 이상적인 구조는 아니지만, 싱글톤 기반 프레임워크의 제약 안에서 가장 실용적인 선택이었다.

## 돌이켜보면

서버 없이 앱 안에서 에이전트를 돌리는 것의 장점은 **도구가 시스템 API에 직접 접근할 수 있다**는 것이다. 전화, 문자, 캘린더, 앱 조작 — 이 모든 것이 네트워크 왕복 없이 즉시 실행된다. 초기 단계 프레임워크의 제약(싱글톤 도구, 직접 히스토리 관리)은 있지만, 네이티브 접근의 이점이 이를 상쇄한다.
