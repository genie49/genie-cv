# 오류났네, 근데 왜 오류난 거지?

프로덕션에서 에러가 터졌다. 로그를 본다. `[ERROR] Tool call failed: timeout`. 그래서? 어떤 사용자가, 어떤 에이전트에서, 몇 초 만에 실패한 건지 알 수 없다. 로그 10줄을 grep으로 조합하며 30분을 쓴 뒤 깨달았다 — **로그가 많은 게 문제가 아니라, 밀도가 낮은 게 문제다.** [loggingsucks.com](https://loggingsucks.com/)의 Wide Event 철학을 도입해 디버깅 시간을 83% 단축한 과정을 정리한다.

## 로그는 쓰기에 최적화되어 있다

전통적 로깅은 코드가 무엇을 하는지를 한 줄씩 기록한다. "요청 시작", "DB 조회", "유저 찾음", "도구 호출", "실패" — 다섯 줄. 개발할 때는 편하다. `logger.info("여기 왔다")`를 찍으면 되니까. 하지만 프로덕션에서 문제를 찾을 때는 이 다섯 줄을 조합해야 한다. 서비스가 여러 개면? 각 서비스 로그를 request ID로 grep해서 수동으로 맞춰본다.

[loggingsucks.com](https://loggingsucks.com/)에서 읽은 문장이 정확히 문제를 짚었다. **"로그는 쓰기에 최적화되어 있지, 조회에 최적화되어 있지 않다."** 디버깅은 고고학(archaeology)이 아니라 분석(analytics)이어야 한다. 로그를 발굴하는 게 아니라, 쿼리를 던져서 답을 바로 얻어야 한다.

실제로 특정 사용자의 AI 응답이 느리다는 리포트를 받았을 때, 전통적 로깅으로 **30분**이 걸렸다. API 로그에서 request ID를 찾고, AI 서버에서 같은 ID를 grep하고, 여러 줄을 조합해서 "어떤 에이전트가 어디서 느렸는지"를 파악했다.

## Wide Event는 요청에 무슨 일이 일어났는지를 하나로 기록한다

Wide Event는 **서비스 홉당 1개 이벤트**를 원칙으로 한다. 그 하나의 이벤트에 요청의 전체 맥락을 담는다. 사용자 ID, 구독 티어, 사용 모델, 응답 시간, 호출된 도구 목록, 서브에이전트 수, 가장 느린 에이전트와 그 소요 시간, 에러 정보 — 15~20개 필드가 한 줄에 들어간다.

같은 문제를 Wide Event로 디버깅하니 **5분**이었다. 한 줄의 이벤트를 열었더니 답이 바로 보였다. 리서치 에이전트가 8.2초를 잡아먹고 있었고, 원인은 웹 검색 도구가 rate limit에 걸려 재시도를 반복한 것이었다. grep을 한 번도 치지 않았다.

여기서 중요한 구분이 있다. **구조화 로깅(JSON 포맷)과 Wide Event는 다르다.** JSON으로 로그를 찍는 건 첫 단계일 뿐이다. Wide Event의 핵심은 기술적 메트릭뿐만 아니라 **비즈니스 컨텍스트까지 담는 것**이다. 사용자의 구독 티어, 사용 중인 모델, 활성화된 기능 플래그 — 이런 정보가 있어야 "프리미엄 유저가 MAX 모델을 쓸 때만 느린 건가?"같은 질문에 답할 수 있다.

## 요청이 시작되면 이벤트가 자란다

Wide Event를 구현하는 패턴은 미들웨어 기반이다.

요청이 시작되면 빈 이벤트 객체를 만든다. 여기에 thread ID, 모델 티어, request ID 같은 기본 컨텍스트를 바인딩한다. 에이전트가 실행되면서 호출된 도구 목록, 서브에이전트 수, 소요 시간이 추가된다. 에러가 발생하면 에러 정보가 붙는다. 요청이 끝나는 시점에 **이 하나의 이벤트를 발행**한다.

코드 곳곳에 `logger.info()`를 찍는 게 아니라, 하나의 이벤트에 컨텍스트를 계속 누적하다가 마지막에 한 번 내보내는 것이다. 이렇게 하면 한 요청에 대한 모든 정보가 하나의 이벤트에 모인다.

이 패턴을 AI 서비스(Python)와 API 서비스(TypeScript) **양쪽 모두에** 적용했다. 언어는 다르지만 철학은 같다. Python은 loguru의 `bind()`로, TypeScript는 pino의 base context로 컨텍스트를 누적한다. 두 서비스의 로그 포맷이 동일하니 로그 수집 파이프라인도 하나로 통합할 수 있었다.

## 서비스 사이를 Request ID가 관통한다

Wide Event가 서비스 하나의 이야기를 담는다면, **Request ID는 서비스를 넘나드는 이야기를 연결**한다. nginx에서 생성된 Request ID가 API 서비스로, 다시 AI 서비스로 전파된다. 한 사용자의 요청이 세 서비스를 거치더라도, 같은 Request ID로 전체 여정을 추적할 수 있다.

응답 헤더에도 Request ID를 포함한다. 사용자가 "이 요청에서 문제가 있었어요"라고 리포트하면, 응답 헤더의 Request ID 하나로 세 서비스의 로그를 한 번에 찾을 수 있다.

## High Cardinality 필드가 핀포인트를 가능하게 한다

Wide Event의 진짜 힘은 **High Cardinality 필드**에서 나온다. 수백만 개의 고유값을 가질 수 있는 필드 — trace ID, user ID, thread ID — 가 있어야 수만 건의 로그에서 특정 요청 하나를 핀포인트할 수 있다.

Low Cardinality 필드(서비스명, 레벨, 모델 티어)로 **범위를 좁히고**, High Cardinality 필드로 **특정 요청을 집어낸다.** "AI 서비스에서 발생한 에러 중, 프리미엄 유저의 MAX 모델 요청에서, user_id가 이것인 건?" — 이런 다차원 쿼리가 가능해진다.

## 민감한 건 로그에도 남기지 않는다

로그에 정보를 많이 담을수록 디버깅은 편해지지만, 보안 리스크도 커진다. 몇 가지 원칙을 정했다. DB 쿼리 로그에서는 파라미터를 일부만 표시하고 나머지는 잘라낸다. Socket.IO 인증 디버깅에서는 토큰 전체가 아니라 **SHA256 해시의 앞 10자만** 로그에 남긴다. 토큰이 같은지 다른지는 확인할 수 있지만, 원본은 노출되지 않는다. 로그가 유출되어도 크레덴셜이 함께 유출되는 상황을 막는 것이다.

배포 버전(커밋 SHA)도 로그에 포함한다. 문제가 발생했을 때 "어떤 배포에서부터 이 에러가 시작됐는지"를 로그만으로 추적할 수 있다.

## 에러는 Discord로 바로 알린다

모든 로그를 보고 있을 수는 없다. 컨테이너 로그를 수집해서, 에러와 경고만 Discord 웹훅으로 보내는 파이프라인을 만들었다. 서비스별로 색상을 다르게 적용해서, 알림을 읽지 않고도 **어떤 서비스 문제인지 0.5초 만에 파악**할 수 있다.

warn 이상만 Discord로 전송하니 전체 로그의 약 5%만 알림으로 보인다. 나머지 95%는 필요 시 조회한다.

## 로그를 읽는 건 사람만이 아니다

한 가지 더 생각하는 것이 있다. 로그를 읽는 주체가 점점 **코딩 에이전트**로 옮겨가고 있다. Claude Code 같은 코딩 에이전트가 에러 로그를 읽고, 원인을 추론하고, 수정 방향을 제안하는 시대다. 그렇다면 로그는 사람이 읽기 좋은 것을 넘어 **AI가 읽기 좋은 구조**여야 한다.

이것도 pino 기반 JSON 로깅을 선택한 이유 중 하나다. `[ERROR] Something went wrong` 같은 비정형 텍스트보다, 필드가 명확히 구분된 JSON이 AI에게 훨씬 파싱하기 쉽다. 에러 타입, 발생 위치, 관련 요청 컨텍스트가 구조화되어 있으면, 코딩 에이전트가 "이 에러는 이 서비스의 이 함수에서, 이 조건일 때 발생한다"를 바로 추론할 수 있다.

## 돌이켜보면

로깅의 목적은 "무슨 일이 있었는지"를 빠르게 파악하는 것이다. 10줄의 산발적 로그보다 **50개 필드를 가진 1개의 Wide Event**가 디버깅에 훨씬 효과적이었다. 로그의 양이 아니라 **밀도**가 중요하다.
