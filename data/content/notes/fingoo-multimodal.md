# 투자 분석 해줄려면 내가 정리해둔 엑셀 정도는 볼 줄 알아야지

"이 엑셀 데이터 분석해줘." 투자자라면 당연히 기대하는 기능이다. 그런데 LLM은 파일 바이너리를 직접 읽지 못한다. 파일을 AI가 이해할 수 있는 형태로 변환하면서도 컨텍스트를 효율적으로 사용하는 파이프라인을 만드는 과정을 정리한다.

## 직접 파싱했다, 그런데 문제가 두 개였다

처음에는 파일 타입별로 파서를 직접 만들었다. CSV는 pandas로, XLSX는 openpyxl로, PDF는 텍스트 추출. 파일이 올라오면 내용을 파싱해서 컨텍스트에 넣었다.

**첫 번째 문제는 컨텍스트 낭비였다.** 1,000행짜리 재무제표 CSV를 파싱해서 통째로 넣으면, 사용자가 "영업이익 추이 알려줘"라고 해도 나머지 999행이 컨텍스트를 잡아먹는다. 불필요한 정보가 너무 많으니 LLM이 핵심을 놓치는 경우도 생겼다. 필요한 행만 추출하는 로직을 추가해봤지만, "어떤 행이 필요한가"를 사전에 판단하는 것 자체가 어려웠다.

**두 번째 문제는 타이밍이었다.** 에이전트가 파일을 **언제 읽어야 하는지**를 잘 못 잡았다. 사용자가 파일을 첨부하고 질문하면, 에이전트가 파일을 먼저 읽어야 할 때가 있고, 다른 도구를 먼저 호출한 뒤에 파일을 참고해야 할 때가 있다. 파싱된 내용을 미리 컨텍스트에 넣어놓으면 타이밍 문제는 없지만, 그러면 첫 번째 문제(컨텍스트 낭비)가 다시 돌아온다. 파싱을 도구로 빼면 컨텍스트는 아끼지만, 에이전트가 적절한 시점에 호출하지 못했다.

두 문제를 동시에 풀어야 했다. **컨텍스트를 아끼면서도, 에이전트가 자연스럽게 파일 내용에 접근할 수 있는 방법.**

## 파싱을 직접 하지 말고 LLM에 위임하자

발상을 바꿨다. 파일 타입별 파서를 직접 만드는 대신, **LLM의 내장 기능에 위임**한다.

Gemini의 File Search Store가 CSV, XLSX, PDF, DOC, HWP를 자동으로 인덱싱한다. 에이전트가 도구를 호출하면서 쿼리를 던지면, File Search Store가 파일에서 관련 부분만 찾아서 돌려준다. 직접 파싱할 때의 두 문제가 동시에 풀렸다. 파일 전체가 아니라 쿼리에 관련된 부분만 반환되니 컨텍스트 낭비가 사라졌고, 에이전트가 "이 파일에서 이걸 찾아줘"라고 쿼리와 함께 요청하니 타이밍 문제도 없어졌다.

금융 데이터의 파일 형식은 천차만별이다. 같은 엑셀이라도 증권사마다 레이아웃이 다르고, PDF 리포트는 표와 텍스트가 섞여 있고, HWP 같은 한국 특화 포맷도 있다. 이 모든 포맷의 파서를 직접 만들고 유지보수하는 것은 현실적이지 않았다. **하나의 API에 위임하는 것**이 핵심 설계 결정이었다.

## 에이전트는 파일 타입을 모른다

에이전트가 파일을 읽는 도구의 인터페이스는 극단적으로 단순하다. **파일 ID와 쿼리, 딱 두 개.** "이 파일에서 영업이익 추이를 찾아줘"라고 호출하면 끝이다. CSV인지 PDF인지 이미지인지, 에이전트는 알 필요가 없다.

도구 안에서는 파일의 MIME 타입을 확인해서 두 갈래로 분기한다.

| 파일 타입 | 처리 경로 | 동작 방식 |
|---|---|---|
| 이미지 (PNG, JPG 등) | Vision 경로 | 이미지 파일과 쿼리를 LLM에 함께 전달. 시각적으로 분석 |
| 문서 (CSV, XLSX, PDF, DOC, HWP) | File Search 경로 | 인덱싱된 파일에서 쿼리와 관련된 부분만 의미 기반 검색 |

두 경로는 **상호 배타적**이다. 업로드 시점에 이미지면 Vision용 저장소에, 문서면 File Search Store에 저장된다. 둘 다에 저장되는 경우는 없다. 도구가 호출되면 어느 저장소에 파일이 있는지 확인하고 해당 경로로 보낸다.

이미지 경로에서는 Vision 모델이 이미지를 직접 본다. 투자자가 다른 플랫폼의 차트 스크린샷을 찍어서 "이거 분석해줘"라고 보내는 경우가 많다. 텍스트로는 추출할 수 없는 시각적 패턴 — 추세선의 기울기, 캔들의 형태, 지지/저항 구간 — 을 AI가 직접 읽어낸다.

문서 경로에서는 File Search Store가 문서를 의미 단위로 인덱싱한 뒤, 쿼리와 관련된 부분만 추출해서 반환한다. 문서 인덱싱은 비동기로 일어나기 때문에, 업로드 직후에는 인덱싱이 끝날 때까지 기다려야 한다. 인덱싱이 안 된 상태에서 도구를 호출하면 "파일을 다시 업로드해주세요"라는 안내가 반환된다 — 에이전트가 이 메시지를 사용자에게 자연스럽게 전달할 수 있도록, **에러도 자연어로 설계**했다.

이 구조의 장점은 **확장성**이다. 새로운 파일 타입을 지원하고 싶으면 파서를 만들 필요 없이, File Search Store가 해당 포맷을 지원하는지만 확인하면 된다. 에이전트 코드는 건드릴 게 없다.

## AI 처리와 사용자 경험을 분리한다

파일은 두 곳에 저장된다. AI가 분석하는 경로(Google APIs)와 사용자가 미리보기/다운로드하는 경로(별도 스토리지)를 분리했다. **Split Storage 전략**이다.

이 분리가 필요한 이유가 있다. AI 처리용 스토리지는 인덱싱, 임베딩, Vision 분석 같은 무거운 작업을 담당한다. 파일이 인덱싱되는 데 시간이 걸릴 수 있다. 이 과정이 사용자의 파일 미리보기를 느리게 만들면 안 된다. 역으로, 프리뷰 이미지를 생성하는 과정이 AI 분석 경로에 영향을 줘서도 안 된다. 두 경로가 독립적이라 한쪽이 느려져도 다른 쪽에 영향을 주지 않는다.

## 48시간 뒤에 사라진다

투자자의 재무 데이터는 민감하다. 포트폴리오 구성, 미공개 실적 데이터, 내부 분석 자료 — 이런 파일을 영구 저장하면 보안 리스크가 커진다.

업로드된 파일은 **48시간 후 자동으로 삭제**된다. 스케줄러가 주기적으로 돌면서 만료된 파일을 AI 처리 스토리지, 프리뷰 스토리지, DB 메타데이터에서 모두 정리한다. 삭제가 한쪽에서 실패해도 나머지는 계속 진행하고, 정리되지 않은 파일은 다음 주기에 다시 시도된다.

48시간이라는 기간은 투자 분석의 맥락에서 정했다. 대부분의 분석 대화는 하루 안에 끝나지만, 다음 날 이어서 질문하는 경우도 있다. 48시간이면 충분한 여유를 주면서도 불필요하게 오래 보관하지 않는 균형점이다.

## 파일 업로드는 공격 벡터다

파일을 받는 기능은 보안 표면이 넓다. 여러 층에서 각각 다른 방식으로 검증한다.

클라이언트에서 파일 형식을 그룹별로 감지하고, 서버에서 MIME 타입을 재확인한다. 한 세션에 첨부할 수 있는 파일 수를 제한한다. 각 파일의 소유자를 기록해서 다른 사용자의 파일에 접근할 수 없게 하고, 삭제 시에도 소유자 확인을 거친다. 만료된 파일은 자동 정리된다. 단일 레이어 방어는 한 곳이 뚫리면 끝이다 — **여러 층에서 각각 독립적으로 검증**하는 것이 원칙이다.

## 돌이켜보면

멀티모달 파이프라인의 핵심 결정은 **"직접 파싱하지 않는다"**였다. 처음에는 파일 타입별 파서를 직접 만들었지만, 컨텍스트 낭비와 타이밍 문제를 동시에 풀 수 없었다. LLM의 내장 기능에 위임하니 두 문제가 함께 풀렸다. 이미지와 문서를 다른 경로로 보내되 에이전트는 그 차이를 모르게 하나의 인터페이스로 추상화한 것, 그리고 AI 처리와 사용자 경험을 스토리지 레벨에서 분리한 것이 설계의 뼈대다.
