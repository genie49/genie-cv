# 스캔 PDF가 검색 가능해지기까지

한국의 발굴조사보고서는 수만 건이 PDF로만 존재한다. 대부분 종이를 스캔한 이미지 PDF라 텍스트 검색 자체가 불가능하다. 게다가 보고서 안에는 유물 사진, 유구 도면, 현장 전경 같은 이미지가 수십~수백 장 포함되어 있는데, 이 이미지의 위치를 파악하는 것도 중요했다. 이 PDF를 텍스트 추출 → 이미지 위치 인식 → 청킹 → 임베딩 → 벡터 DB 저장까지 End-to-End로 처리하는 파이프라인을 정리한다.

## 텍스트만 뽑으면 되는 줄 알았다

처음에는 단순했다. PDF에서 텍스트를 뽑고, 청킹하고, 임베딩하면 끝이라고 생각했다. PyMuPDF로 텍스트를 추출하면 수백 페이지 PDF도 수 초 만에 처리된다. GPU도 필요 없다.

그런데 발굴조사보고서는 대부분 **스캔 PDF**였다. `page.get_text()`를 호출해도 빈 문자열만 돌아왔다. 텍스트 레이어가 없으니까. OCR이 필수였다.

OCR만 있으면 된다고 생각해서 처음에는 텍스트 추출 전용 OCR을 적용했다. 텍스트는 뽑혔다. 하지만 곧 다른 문제가 보였다. 보고서 안의 **이미지 위치를 알 수 없었다.** 유물 사진이 몇 페이지에 있는지, 도면이 어디에 있는지 — 텍스트만 뽑는 OCR로는 파악할 수 없었다. 이미지 검색 기능을 만들려면 이미지의 위치를 알아야 했다.

## 텍스트도 뽑고 이미지 위치도 잡는 OCR

필요한 건 단순 텍스트 OCR이 아니라 **문서 구조 인식(Document Parsing)**이었다. 페이지를 의미 단위 블록으로 분류하고, 텍스트 블록은 마크다운으로, 표 블록은 표로, 이미지 블록은 **위치 좌표(bbox)**를 반환하는 시스템.

PaddleOCR의 Document Parser가 이걸 해결했다. 3단계 파이프라인으로 동작한다. 먼저 페이지의 레이아웃을 분석해서 블록을 분류하고, 텍스트/표/수식 블록은 마크다운으로 변환하고, 이미지 블록은 bbox 좌표만 추출한다. 결과물은 마크다운 텍스트와 이미지 위치 목록이 함께 나온다.

이 bbox 좌표가 핵심이다. 나중에 원본 PDF 페이지에서 해당 좌표로 이미지를 잘라내고, Gemini Vision으로 설명을 생성하고, 벡터화해서 이미지 검색을 가능하게 만든다.

## 모든 PDF에 OCR을 돌리면 비용이 폭발한다

PaddleOCR는 GPU가 필요하다. 모든 PDF에 OCR을 돌리면 GPU 비용이 불필요하게 높아진다. 텍스트 레이어가 있는 PDF는 PyMuPDF로 수 초 만에 처리할 수 있는데, 굳이 GPU를 쓸 이유가 없다.

**조건부 폴백**으로 해결했다. PyMuPDF로 먼저 텍스트를 추출하고, 페이지당 평균 문자 수가 100자 미만이면 OCR로 넘긴다. 발굴조사보고서는 대부분 스캔 PDF라 OCR로 직행하는 경우가 많지만, UNESCO 보고서 같은 디지털 PDF는 PyMuPDF에서 끝난다.

100자라는 임계값은 실제 데이터에서 실험적으로 결정했다. 스캔 PDF에서 PyMuPDF가 뽑아내는 문자는 페이지 번호나 머리글 정도라 보통 10~30자 수준이고, 텍스트 PDF는 수백 자가 나온다. 100자면 두 경우를 안정적으로 구분할 수 있었다.

## GPU는 쓸 때만 켠다

OCR 서비스는 서버리스 GPU(Modal) 위에서 실행된다. 요청이 없으면 GPU 인스턴스가 자동으로 꺼져서 비용이 발생하지 않는다.

VLM 모델을 vLLM 서버로 띄우고, PaddleOCR 파이프라인이 내부적으로 이 서버를 호출하는 구조다. 레이아웃 분석은 CPU에서, VLM 추론만 GPU에서 처리한다. KV 캐시를 fp8로 줄여 메모리를 절약하고, 컨테이너 시작 시 모델을 미리 로드해서 이후 요청은 warm 상태에서 처리된다. 모델 가중치는 볼륨에 캐시해서 콜드 스타트 시에도 다운로드를 건너뛴다.

마지막 요청 후 2분간 유휴 상태가 지속되면 컨테이너가 자동 종료된다. 보고서 수백 건을 일괄 처리하는 초기 인제스션 이후에는 신규 보고서가 등록될 때만 간헐적으로 GPU가 켜진다. **안 쓸 때 비용이 0**인 것이 서버리스 GPU의 진짜 가치다.

## 이미지는 잘라내고, 인접한 건 합친다

OCR이 반환한 bbox 좌표로 페이지 이미지에서 해당 영역을 잘라낸다. 그런데 하나의 유물 사진이 여러 블록으로 분리되는 경우가 있다. OCR이 사진의 윗부분과 아랫부분을 다른 블록으로 인식한 것이다.

**Union-Find 알고리즘**으로 인접하거나 겹치는 bbox를 그룹핑해서 합친다. 페이지 높이의 2%를 임계값으로, 이 거리 안에 있는 bbox들을 하나의 그룹으로 묶는다. 텍스트가 거의 없는 페이지(50자 미만)는 사진 도판 페이지로 판단해서 개별 crop 대신 **페이지 전체를 하나의 이미지로 저장**한다.

잘라낸 이미지에는 Gemini Vision으로 설명을 생성한다. 이때 **주변 텍스트 컨텍스트를 함께 제공**하는 것이 중요하다. 이미지만 보내면 "토기 사진"이지만, 보고서 본문을 함께 보내면 "OO 유적 3호 주거지에서 출토된 경질무문토기"처럼 맥락이 포함된 설명이 나온다.

## 텍스트는 청킹하고 페이지를 매핑한다

추출된 텍스트는 2,000자 단위로 청킹한다. 500자 오버랩으로 문맥이 끊기지 않게 하고, 문장 경계에서 자른다. 각 청크에는 원본 PDF의 시작/종료 페이지 번호가 매핑된다. 검색 결과에서 "이 내용은 보고서 42~44페이지에 있습니다"라고 안내할 수 있는 근거다.

청크마다 보고서의 핵심 메타데이터 — 허가번호, 시대, 지역, 유적 유형 — 가 함께 저장된다. 데이터 정규화 관점에서는 낭비지만, 벡터 DB에는 JOIN이 없다. 청크 레벨에서 바로 필터링하려면 **비정규화가 오히려 정답**이다.

## 돌이켜보면

파이프라인은 세 번 바뀌었다. PyMuPDF만 쓰던 시절, 텍스트 전용 OCR을 붙인 시절, 그리고 문서 구조 인식 OCR로 전환한 현재. 매번 "이제 됐다"고 생각했지만, 다음 요구사항(이미지 위치 인식)이 구조를 다시 바꿨다. 결국 **텍스트와 이미지를 동시에 다루는 파이프라인**이 되어야 했다.
