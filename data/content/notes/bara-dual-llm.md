# 클라우드가 안 되면? 폰이 직접 생각한다

지하철 터널, 비행기 안, 해외 로밍 꺼진 상태. 인터넷이 없으면 AI 비서가 먹통이 된다. 클라우드 LLM을 기본으로 사용하되, 온디바이스 LLM으로 전환할 수 있는 이중 구조를 설계한 과정을 정리한다.

## 클라우드가 기본이다

현재는 Gemini API를 기본 LLM으로 사용한다. 무료 티어로도 분당 15회 요청이 가능해서, 개인 음성 비서 수준의 트래픽에는 충분하다. 응답 품질도 안정적이다.

하지만 클라우드 의존은 본질적 한계가 있다. 네트워크가 끊기면 에이전트가 동작하지 않는다. 음성 파이프라인의 나머지 — 웨이크워드, STT, TTS — 는 전부 온디바이스인데, LLM만 클라우드에 의존하고 있다.

## 온디바이스 LLM의 가치는 비용 절감이 아니다

온디바이스 LLM을 도입하려는 이유를 처음에는 비용 절감이라고 생각했다. 하지만 계산해보면 Gemini 무료 티어만으로 충분하다. **진짜 가치는 가용성과 프라이버시**다.

**가용성** — 네트워크 없이도 에이전트가 동작한다. 지하철, 비행기, 오지 — 인터넷이 안 되는 환경에서도 전화를 걸고, 일정을 잡고, 알림을 읽을 수 있다.

**프라이버시** — 사용자의 음성 명령이 서버를 거치지 않는다. "엄마한테 전화해줘"라는 개인적인 명령이 클라우드에 전송되지 않는다.

## Gemma 3n — 폰에서 돌릴 수 있는 LLM

Google의 Gemma 3n이 온디바이스 후보다. E2B(Embeddings to Bits) 아키텍처로 설계되어, 모바일 환경에 최적화되어 있다.

핵심 기술 세 가지가 있다. **Per-Layer Embeddings**은 레이어별로 필요한 파라미터만 활성화해서 메모리 사용을 줄인다. **NPU 최적화**는 모바일 칩의 Neural Processing Unit에서 효율적으로 돌아가도록 양자화를 적용한다. **선택적 파라미터 활성화**는 입력에 따라 필요한 파라미터만 활성화해서 추론 속도를 높인다.

## 온디바이스 Tool Calling이 진짜 난관이다

LLM을 폰에서 돌리는 것 자체보다, **도구 호출을 제대로 하게 만드는 것**이 더 어렵다.

클라우드 Gemini는 Function Calling을 네이티브로 지원한다. JSON 형태로 도구 정의를 주면, 모델이 호출할 도구와 인자를 구조화된 형태로 반환한다. 온디바이스 모델은 이 기능이 제한적이다. 모델 출력을 직접 파싱해서 도구 호출로 변환해야 한다.

단일 도구 호출은 정확도가 높지만, 2단계 이상의 연쇄 호출 — "연락처 검색 → 전화 걸기" — 에서 정확도가 급격히 떨어진다. 클라우드와 온디바이스의 **품질 차이를 사용자가 인지하지 않는 수준**으로 만드는 것이 목표다.

## 자동 전환 전략

네트워크 상태에 따라 클라우드와 온디바이스를 자동으로 전환하는 하이브리드 엔진을 설계했다. 기본은 클라우드, 네트워크 불가 시 온디바이스로 폴백한다. 사용자 설정으로 "항상 온디바이스"를 선택할 수도 있다.

에이전트 인터페이스가 추상화되어 있어서, 어떤 LLM을 쓰든 도구 실행 로직은 동일하다. 전환 시 대화 히스토리도 유지된다.

## 돌이켜보면

이중 LLM 구조의 핵심은 **"선택할 수 있다"**는 것이다. 클라우드가 되면 클라우드를, 안 되면 온디바이스를. 사용자가 프라이버시를 원하면 온디바이스를. 하나의 경로만 있으면 그 경로가 막혔을 때 끝이지만, 두 경로가 있으면 하나가 막혀도 동작한다.
