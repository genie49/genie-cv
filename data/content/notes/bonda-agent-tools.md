# 에이전트에게 문화재 전문가의 도구를 쥐어주면

"청동기시대 주거지 특징이 뭐야?" 단순 키워드 검색으로도 답할 수 있다. 하지만 보고서를 시맨틱 검색하고, 관련 이미지를 찾고, 전문 용어의 정의를 사전에서 확인한 뒤, 이 모든 결과를 종합해 답변을 구성하려면? 에이전트가 도구를 직접 선택하고 조합해야 한다.

## 4개 도구, 에이전트가 고른다

에이전트에게 4개의 특화 도구를 줬다. 각 도구는 검색 대상과 방식이 다르다.

**보고서 검색** — 발굴조사보고서와 UNESCO 보고서를 시맨틱 검색, 메타데이터 필터, 또는 둘의 조합으로 검색한다. 에이전트가 질문의 성격에 따라 검색 모드를 직접 선택한다. "경기도 보고서 목록"에는 필터만, "청동기시대 주거지 구조"에는 시맨틱 검색을, "전남 삼국시대 고분"에는 둘을 조합한다.

**이미지 검색** — 유물 사진, 유구 도면을 텍스트로 찾거나, 특정 이미지와 유사한 이미지를 찾는다. 텍스트 검색과 이미지 유사도 검색이 하나의 도구에 모드로 분리되어 있다.

**용어 사전** — 13,000개 이상의 고고학 용어와 유적 정의를 검색한다. 에이전트가 답변에 전문 용어를 사용할 때, 이 도구로 정의를 확인하고 출처 사전명까지 인용한다. "송국리형 주거지"가 뭔지 정확히 알고 답하는 것과 모르고 답하는 것은 품질이 다르다.

**현재 시간** — 단순하지만 필요한 도구다. 시스템 프롬프트에서 현재 시점 맥락을 제공한다.

"경기도 청동기시대 주거지에서 출토된 토기 사진"이라는 하나의 질문에, 에이전트가 보고서 검색 → 이미지 검색 → 용어 사전을 자동으로 조합한다. 어떤 도구를 어떤 순서로 쓸지를 사람이 정하지 않고, 에이전트가 질문을 분석해서 판단한다.

## 검색 결과에는 출처가 따라붙는다

검색 결과를 그냥 텍스트로 보여주면 "AI가 말했으니 맞겠지"가 된다. 연구자에게는 **출처가 필수**다. "이 내용은 어떤 보고서의 몇 페이지에 있는가"를 알아야 원본을 확인할 수 있다.

보고서 검색 결과에는 허가번호, 보고서명, 페이지 범위가 함께 표시된다. 이미지 검색 결과에는 해당 이미지가 포함된 보고서와 페이지가 표시된다. 용어 사전 결과에는 출처 사전명이 표시된다. 에이전트가 답변을 구성할 때 이 출처 정보를 자연스럽게 인용하도록 시스템 프롬프트에서 가이드한다.

프론트엔드에서는 이 출처를 **클릭 가능한 링크**로 렌더링한다. 보고서 검색 결과를 클릭하면 해당 보고서의 상세 페이지로 이동하고, 이미지를 클릭하면 원본 이미지를 볼 수 있다.

## 이미지를 드래그해서 다음 질문에 쓴다

여기서 한 걸음 더 나아간 것이 있다. 검색 결과가 단순히 보여주고 끝나는 게 아니라, **다음 대화의 컨텍스트로 다시 사용**된다.

이미지 검색 결과는 채팅 옆 패널에 그리드로 표시된다. 각 이미지 카드에는 설명, 페이지 번호, 유사도 점수가 함께 보인다. 여기서 핵심 UX가 있다 — 이미지 카드에 마우스를 올리면 **"드래그해서 참조 추가"** 힌트가 나타난다. 이미지를 채팅 입력창에 드래그 앤 드롭하면, 해당 이미지의 참조가 메시지에 자동 삽입된다.

"이 토기랑 비슷한 거 더 찾아줘"라고 입력하면, 에이전트가 드롭된 이미지의 ID를 인식해서 유사 이미지 검색을 실행한다. 기존 이미지의 벡터와 가장 가까운 벡터를 찾는 것이라 별도의 임베딩 계산이 필요 없다 — 이미 저장된 벡터를 그대로 활용한다.

사용자가 보고서 검색 결과를 보고 "이 보고서에서 출토된 유물 사진 보여줘"라고 이어서 질문하면, 에이전트는 대화 히스토리에서 보고서 ID를 컨텍스트로 활용해서 해당 보고서의 이미지를 검색한다.

**검색 → 결과 확인 → 드래그 앤 드롭 → 후속 질문 → 재검색** — 이 루프가 자연스럽게 돌아가는 게 핵심이다. 연구자는 하나의 질문으로 끝나지 않는다. 결과를 보고, 방향을 좁히고, 다시 질문한다. 이 과정에서 검색 결과를 직접 잡아서 다음 질문에 넣을 수 있다는 게 텍스트만으로 대화하는 것과 다른 점이다.

## LLM 친화적 인터페이스 설계

도구의 파라미터를 LLM이 안정적으로 생성할 수 있는 형태로 설계하는 것이 중요했다. 시대나 유적 유형처럼 복수 값이 필요한 필드는 배열이 아닌 **쉼표 구분 문자열**로 받는다. LLM이 배열보다 문자열을 더 안정적으로 생성하기 때문이다. 내부에서 파싱한다.

이미지 검색 결과는 텍스트 스트리밍 중간에 끼우지 않는다. 에이전트가 텍스트 답변을 생성하는 동안 이미지 JSON이 중간에 섞이면 프론트엔드 렌더링이 복잡해진다. 텍스트 스트리밍이 완료된 뒤 **별도 이벤트로 이미지 결과를 발행**한다. 프론트엔드가 텍스트와 이미지를 독립적으로 렌더링할 수 있고, 이미지 로딩이 텍스트 표시를 블로킹하지 않는다.

## 모델을 바꿔도 대화가 이어진다

하나의 에이전트 코드로 여러 LLM을 전환할 수 있다. 사용자가 UI에서 모델을 바꾸면 다음 요청부터 해당 모델이 사용된다. 대화 히스토리는 서버에 두지 않고 **프론트엔드가 관리**하기 때문에, 모델을 중간에 전환해도 맥락이 유실되지 않는다. 서버 재시작에도 마찬가지다.

이 Stateless 설계의 장점은 모델 전환뿐 아니다. Cloud Run이 서버리스로 스케일링될 때 어떤 인스턴스가 요청을 받아도 동일하게 동작한다. 세션 상태가 특정 인스턴스에 묶여 있지 않으니까.

모델별 에이전트 인스턴스는 캐싱해서, 같은 모델 요청은 재초기화 없이 즉시 처리한다.

## 돌이켜보면

에이전트의 가치는 도구 하나하나가 아니라 **도구의 조합과 결과의 연속성**에서 나왔다. 보고서 검색만으로도 답할 수 있지만, 이미지 검색과 용어 사전을 함께 쓰면 답변의 깊이가 달라진다. 그리고 그 결과가 다음 질문의 출발점이 되는 흐름 — 검색, 확인, 좁히기, 재검색 — 이 연구자의 실제 탐색 방식과 맞닿아 있다.
