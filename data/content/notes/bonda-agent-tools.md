# 에이전트에게 문화재 전문가의 도구를 쥐어주면

"청동기시대 주거지 특징이 뭐야?" 단순 키워드 검색으로도 답할 수 있다. 하지만 보고서를 시맨틱 검색하고, 관련 이미지를 찾고, 전문 용어의 정의를 사전에서 확인한 뒤, 이 모든 결과를 종합해 답변을 구성하려면? 에이전트가 도구를 직접 선택하고 조합해야 한다.

## 4개 도구, 에이전트가 고른다

에이전트에게 4개의 특화 도구를 줬다.

**보고서 검색** — 발굴조사보고서와 UNESCO 보고서를 시맨틱 검색, 메타데이터 필터, 또는 둘의 조합으로 검색한다. 에이전트가 질문의 성격에 따라 검색 모드를 직접 선택한다. "경기도 보고서 목록"에는 필터만, "청동기시대 주거지 구조"에는 시맨틱 검색을, "전남 삼국시대 고분"에는 둘을 조합한다.

**이미지 검색** — 유물 사진, 유구 도면을 텍스트로 찾거나, 특정 이미지와 유사한 이미지를 찾는다. 텍스트 검색과 이미지 유사도 검색이 하나의 도구에 모드로 분리되어 있다.

**용어 사전** — 13,000개 이상의 고고학 용어와 유적 정의를 검색한다. 에이전트가 답변에 전문 용어를 사용할 때, 이 도구로 정의를 확인하고 출처 사전명까지 인용한다. "송국리형 주거지"가 뭔지 정확히 알고 답하는 것과 모르고 답하는 것은 품질이 다르다.

**현재 시간** — 단순하지만 필요한 도구다. 시스템 프롬프트에서 현재 시점 맥락을 제공한다.

"경기도 청동기시대 주거지에서 출토된 토기 사진"이라는 하나의 질문에, 에이전트가 보고서 검색 → 이미지 검색 → 용어 사전을 자동으로 조합한다.

## LLM 친화적 인터페이스 설계

도구의 파라미터를 LLM이 안정적으로 생성할 수 있는 형태로 설계하는 것이 중요했다. 시대나 유적 유형처럼 복수 값이 필요한 필드는 배열이 아닌 **쉼표 구분 문자열**로 받는다. LLM이 `"삼국시대,백제시대"` 형태로 안정적으로 생성하고, 내부에서 파싱한다.

이미지 검색 결과는 텍스트 스트리밍 중간에 끼우지 않는다. 이미지 데이터를 텍스트 스트림에 섞으면 프론트엔드 렌더링이 복잡해지기 때문이다. 텍스트 스트리밍이 끝난 뒤 **별도 이벤트로 이미지 결과를 발행**한다. 프론트엔드가 텍스트와 이미지를 독립적으로 렌더링할 수 있다.

## 모델을 바꿔도 대화가 이어진다

하나의 에이전트 코드로 여러 LLM을 전환할 수 있다. 사용자가 UI에서 모델을 바꾸면 다음 요청부터 해당 모델이 사용된다. 대화 히스토리는 서버에 두지 않고 **프론트엔드가 관리**(Stateless)하기 때문에, 모델을 중간에 전환해도 맥락이 유실되지 않는다. 서버 재시작에도 마찬가지다.

모델별 에이전트 인스턴스는 캐싱해서, 같은 모델 요청은 재초기화 없이 즉시 처리한다.

## 돌이켜보면

에이전트의 가치는 도구 하나하나가 아니라 **도구의 조합에서** 나왔다. 보고서 검색만으로도 답할 수 있지만, 이미지 검색과 용어 사전을 함께 쓰면 답변의 깊이가 달라진다. 어떤 도구를 어떤 순서로 조합할지를 LLM이 판단하게 한 것이 핵심이다.
