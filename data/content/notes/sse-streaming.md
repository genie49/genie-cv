# 답변 아래에 출처 칩이 붙는다

AI 챗봇의 응답을 토큰 단위로 실시간 스트리밍하고, 검색 결과를 내부 페이지 링크로 자동 변환하는 인용 시스템을 구현한 과정을 정리한다.

## SSE 표준 대신 줄 단위 JSON 스트리밍

LLM 응답을 스트리밍하는 방식으로 SSE 표준(`text/event-stream`)이 아닌 **줄 단위 JSON(Newline-Delimited JSON)** 스트리밍을 선택했다. JSON 객체를 한 줄에 하나씩, 줄바꿈으로 구분해서 보내는 포맷이다. 한 줄을 받을 때마다 바로 파싱할 수 있어서 실시간 스트리밍에 적합하다.

SSE 표준의 `EventSource` API는 GET 요청만 지원한다. 채팅 메시지와 히스토리를 요청 본문으로 보내야 하는데 GET으로는 불가능하다. NDJSON은 일반 `fetch` + POST로 요청하고, `ReadableStream` reader로 줄 단위 파싱한다. 커스텀 헤더, abort signal도 자유롭게 쓸 수 있다.

스트림에는 세 가지 이벤트 타입이 흐른다.

| 타입 | 시점 | 내용 |
|---|---|---|
| `token` | 스트리밍 중 | 텍스트 청크 |
| `citations` | 스트리밍 완료 후 | 인용 배열 (출처 정보) |
| `done` | 마지막 | 스트림 종료 신호 |

에러가 발생하면 `error` 타입으로 분류된 한국어 메시지가 전송된다.

## 에이전트는 인용 번호를 쓰지 않는다

일반적인 RAG 시스템은 LLM에게 `[1]`, `[2]` 같은 인용 번호를 응답에 포함하도록 지시한다. 이 챗봇은 반대로 갔다. **시스템 프롬프트에서 인용 번호 사용을 금지**한다.

이유는 인용 시스템을 에이전트가 아니라 **코드가 제어**하기 위해서다. LLM이 인용 번호를 직접 쓰면 번호가 틀리거나, 없는 출처를 인용하거나, 포맷이 일관되지 않을 수 있다. 대신 검색 도구가 반환한 결과를 서버가 자동으로 인용 데이터로 변환하고, 클라이언트가 렌더링한다. LLM은 자연스러운 문장만 생성하면 된다.

## 소스 경로가 내부 라우트로 자동 변환된다

RAG 검색 결과에는 소스 파일 경로가 포함된다. 이 경로를 포트폴리오 사이트의 내부 라우트로 변환해야 한다.

정적 페이지(about.md, experience.md 등)는 하드코딩된 매핑 테이블로 처리한다. 개발 노트는 파일 경로 패턴에서 프로젝트 슬러그와 노트 ID를 추출해서 라우트를 생성한다. **새 개발 노트를 추가해도 코드 변경 없이 인용 라우트가 자동으로 동작**하는 구조다.

변환된 라우트는 **projects.json과 대조해서 유효성을 검증**한다. 존재하지 않는 프로젝트나 삭제된 노트의 인용이 생성되지 않도록 필터링한다. 깨진 링크가 사용자에게 노출되는 것을 방지하는 안전장치다.

## 인용은 메시지 하단에 칩으로 표시된다

인용 데이터는 텍스트에 인라인으로 삽입되지 않는다. 응답 텍스트 아래에 **Citation Chips**로 렌더링된다. 각 칩에는 출처 유형에 따른 아이콘(프로젝트, 파일, 사용자 등)과 라벨이 표시되고, 클릭하면 React Router의 클라이언트 사이드 라우팅으로 **페이지 새로고침 없이** 해당 페이지로 이동한다.

기술적으로는 인용 데이터를 HTML 코멘트로 텍스트 끝에 삽입하고, 렌더링 컴포넌트가 이 코멘트를 파싱해서 칩으로 변환한다. 마크다운 본문과 인용 메타데이터를 하나의 문자열로 전달하면서도 렌더링은 분리하는 방식이다.

## 돌이켜보면

인용 시스템의 핵심 결정은 **"LLM에게 인용을 맡기지 않는다"**였다. 에이전트는 자연스러운 문장만 생성하고, 인용은 코드가 자동으로 생성하고 검증한다. 소스 경로 → 라우트 변환이 규칙 기반이라 새 콘텐츠 추가 시 코드 변경이 필요 없고, projects.json 검증으로 깨진 링크를 방지한다.
