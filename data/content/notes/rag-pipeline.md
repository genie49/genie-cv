# RAG 파이프라인 구축기

LanceDB와 Gemini Embedding을 활용한 RAG 파이프라인 설계 및 구현 과정을 정리합니다.

## 왜 RAG인가?

포트폴리오 챗봇에서 LLM이 개인 데이터를 정확하게 답변하려면 외부 지식 검색이 필수입니다. Fine-tuning 대신 RAG를 선택한 이유는 데이터 업데이트가 잦고, 소스 인용이 중요하기 때문입니다.

## 임베딩 파이프라인

### 데이터 소스
- 마크다운 파일: 자기소개, 학력, 경력, 프로젝트 설명, 개발 노트
- Mermaid 파일: 아키텍처 다이어그램 (텍스트로 임베딩)
- JSON 파일: Q&A 데이터 (텍스트 변환 후 임베딩)

### 청크 분할
마크다운을 H2 헤딩 기준으로 분할하고, 각 청크에 소스 파일 경로와 프로젝트 슬러그 메타데이터를 포함합니다.

### 벡터 저장
Gemini Embedding API로 벡터화한 후 LanceDB에 저장합니다. LanceDB는 서버리스로 별도 인프라 없이 파일 기반으로 동작합니다.

## 검색 흐름

1. 사용자 질문을 Gemini Embedding으로 벡터화
2. LanceDB에서 코사인 유사도 기반 top-k 검색
3. 검색된 청크를 LLM 컨텍스트로 전달
4. 인용 소스를 프론트엔드 라우트에 매핑

## 인사이트

- 청크 크기가 너무 크면 검색 정확도가 떨어지고, 너무 작으면 컨텍스트가 부족합니다
- 메타데이터(projectSlug 등)를 임베딩 시 함께 저장하면 인용 매핑이 간편합니다
- LanceDB는 소규모 데이터에 적합하며, 별도 서버 운영이 필요 없어 개인 프로젝트에 이상적입니다
