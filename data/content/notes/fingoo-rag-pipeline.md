# AI가 어제 뉴스도 모르면서 투자 분석을 한다고?

"어제 삼성전자 실적 발표 봤어?" LLM에게 물으면 모른다. 학습 데이터에 없으니까. 투자 분석 AI가 최신 시장 상황을 모르면 쓸모가 없다. 매일 경제 데이터를 수집해 Knowledge Base를 채우고, 부족하면 실시간 웹 검색으로 보완하는 2단계 파이프라인을 만든 과정을 정리한다.

## 매번 웹 검색하면 비용이 폭발한다

단순한 접근은 모든 질문에 웹 검색을 붙이는 것이다. 정확하긴 하지만, 하루 500쿼리면 월 $150 이상이 든다. 반대로 벡터 DB 검색만 쓰면 관련 데이터가 없을 때 "모르겠습니다"가 된다.

**2단계 파이프라인**으로 해결했다. 먼저 Knowledge Base(KB)를 검색하고, 결과가 충분한지 LLM이 판정한다. 충분하면 KB 결과만 사용하고, 불충분하면 웹 검색으로 보완한다. 이 구조로 웹 검색 호출을 **80% 줄여** 월 API 비용이 $150에서 $30으로 떨어졌다.

판정 모델로 경량 모델을 선택했다. "이 검색 결과가 질문에 답할 수 있는가?"는 단순 Yes/No 판단이라 고성능 모델이 필요 없다. 판정 시간 평균 500ms, 비용은 주력 모델 대비 1/10.

## Knowledge Base는 매일 채운다

KB를 채우는 건 매일의 크롤링이다. 글로벌 시장 데이터, 국내 시총 상위 종목, 재무제표, 경제 지표 — 여러 소스에서 수집한 데이터를 임베딩해서 벡터 DB에 저장한다. 크롤링은 시장 마감 후 하루 한 번 돌리고, 장중 실시간 데이터는 웹 검색이 담당한다.

수집된 데이터는 벡터 DB에만 들어가는 게 아니다. **매일 시황 요약을 자동 생성하는 스케줄러**도 함께 돌린다. 크롤링이 완료되면 당일 수집된 데이터를 LLM에 넘겨 시장 동향, 주요 종목 변동, 경제 지표 변화를 요약한 시황 리포트를 생성한다. 이 리포트는 사용자가 에이전트에게 질문하지 않아도 **앱에 접속하면 바로 볼 수 있는 형태**로 제공된다. 에이전트를 거치면 응답까지 수 초가 걸리지만, 미리 생성된 시황 요약은 즉시 조회할 수 있다. 매일 반복되는 "오늘 시장 어때?" 같은 질문을 에이전트가 매번 처리하지 않아도 되니 API 호출 비용도 추가로 절감됐다.

벡터 인덱스로 HNSW를 선택한 이유가 있다. 매일 크롤링 데이터가 추가되는 환경에서 IVFFlat은 데이터를 추가할 때마다 리인덱싱이 필요하다. HNSW는 동적으로 데이터를 추가해도 검색 품질이 유지된다. 운영 부담이 완전히 다르다.

## "삼성전자"를 검색하면 "삼성SDI"가 나왔다

임베딩 모델이 "삼성전자"와 "삼성SDI"를 의미적으로 유사한 벡터로 매핑했다. 둘 다 삼성 그룹사고 반도체/전자 분야라 코사인 유사도가 높게 나온다. 기업 특정 검색 정확도가 **78%**에 불과했다.

임베딩 파인튜닝은 효과적이지만 데이터셋 구축 비용이 너무 컸다. 대신 **메타데이터 필터링**으로 해결했다. 임베딩 저장 시 기업 코드를 메타데이터로 함께 저장하고, 검색 시 기업 코드로 필터를 건다. 에이전트가 먼저 심볼 검색 도구로 정확한 기업 코드를 확인한 뒤, 해당 코드로 KB를 검색하는 2단계 프로세스다.

정확도가 78%에서 **95%**로 올라갔다. 파인튜닝 대비 구현 비용은 1/100.

## 웹 검색이 실패해도 멈추지 않는다

외부 API는 언제든 실패할 수 있다. Rate limit도 빈번하다. 웹 검색이 실패하면 KB 결과가 있는지 확인하고, 있으면 그것만으로 응답한다. 완벽하지 않더라도 "아무것도 못 찾았습니다"보다는 낫다. **Graceful Degradation** — 한 층이 무너져도 다음 층이 받치는 구조다.

## 돌이켜보면

2단계 파이프라인의 핵심은 **"먼저 가진 걸 확인하고, 없으면 찾으러 간다"**였다. KB 검색 → LLM 판정 → 필요시 웹 검색. 이 구조가 비용 80% 절감과 품질 유지를 동시에 가능하게 했다.
