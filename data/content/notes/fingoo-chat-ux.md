# AI야 숫자 나열만 해주면 보기가 힘들어...

"삼성전자 분석해줘"라고 물으면 AI가 PER 12.3배, PBR 1.1배, ROE 8.7%... 숫자를 줄줄 읊는다. 맞는 말이긴 한데, **읽기가 힘들다.** 투자자는 숫자를 텍스트로 읽고 싶은 게 아니라 차트로 보고 싶다. 추세를 보고 싶고, 비교하고 싶고, 직접 만져보고 싶다.

금융 AI 채팅에서 텍스트만으로는 부족했다. 대화 안에서 차트가 그려지고, 지표가 추가되고, 사용자가 직접 조작할 수 있는 경험을 만드는 과정을 정리한다.

## AI가 말하면서 차트를 그린다

에이전트가 응답을 생성하는 동안 네 가지가 동시에 흐른다.

| 흐름 | 무엇이 일어나는가 | 사용자가 보는 것 |
|---|---|---|
| 텍스트 스트리밍 | LLM 토큰이 한 글자씩 도착 | 채팅 버블에 타이핑 효과 |
| 도구 실행 상태 | 서브에이전트가 도구를 호출 | "기술 분석 중", "시장 데이터 조회 중" |
| 시각화 커맨드 | AI가 차트 생성 명령을 내림 | 채팅 안에 차트가 나타남 |
| 인용 정보 | 검색 결과의 출처가 매핑됨 | 텍스트에 클릭 가능한 출처 링크 |

텍스트만 스트리밍하는 일반 채팅과 다른 점은, **차트가 대화의 일부**라는 것이다. AI가 "삼성전자의 최근 6개월 주가 추이를 보면"이라고 말하는 동시에, 바로 아래에 라인차트가 나타난다. 캔들스틱, 트리맵, 파이차트, 바차트 — AI가 분석 내용에 맞는 차트 타입을 선택해서 커맨드를 보낸다.

여기서 문제가 하나 있었다. 차트 커맨드가 스트리밍 도중에 도착하면, 텍스트가 쌓이는 와중에 차트가 갑자기 나타났다 사라졌다를 반복한다. **깜빡임.**

게임 엔진의 **이중 버퍼링(Double Buffering)** 패턴을 적용했다. 스트리밍 중에는 차트 커맨드를 임시 버퍼에 쌓아두고, 응답이 완료되면 버퍼를 한 번에 확정한다. 사용자는 스트리밍이 끝난 뒤 차트가 깔끔하게 나타나는 경험을 한다.

## 도구 7개의 계층을 보여준다

Supervisor가 서브에이전트를 호출하고, 서브에이전트가 전문 도구를 호출한다. 이 **부모-자식 관계**를 UI에서 트리 형태로 보여준다. "기술 분석 에이전트"가 실행되면 그 아래에 "RSI 계산", "MACD 분석" 같은 자식 도구가 들여쓰기되어 표시된다.

스트리밍 중 텍스트가 1초 이상 멈추면 "분석 중..." 상태로 전환한다. AI가 내부적으로 도구를 실행하거나 서브에이전트 응답을 기다리는 중일 수 있기 때문이다. 아무런 피드백 없이 멈추면 사용자는 "고장났나?" 생각하지만, 상태가 표시되면 기다릴 수 있다.

## 데이터에는 출처가 있어야 한다

금융 도메인에서 "AI가 말했으니 맞겠지"는 통하지 않는다. 투자 판단에 영향을 주는 정보에는 **출처가 필수**다.

AI가 응답을 생성할 때 검색 결과를 인용하면, 해당 텍스트 구간에 인용 마커가 붙는다. 프론트엔드가 이 마커를 파싱해서 클릭 가능한 출처 링크로 변환한다. 사용자는 "삼성전자 PER이 업종 평균 대비 저평가"라는 문장을 보고, 바로 옆의 출처를 클릭해서 원본 데이터를 확인할 수 있다.

인용 시스템은 기술적으로는 텍스트 파싱이지만, 비즈니스적으로는 **서비스 신뢰의 기반**이다.

## 보는 것과 바꾸는 것은 다르다

AI가 "삼성전자 차트에 볼린저밴드를 추가할까요?"라고 물으면, 사용자가 확인한 뒤에 차트가 바뀐다. 반면 "현재 차트 상태를 확인할게요"는 확인 없이 바로 실행된다.

이 구분은 Unix의 **read/write 권한 모델**에서 착안했다. 파일을 읽는 건 안전하지만 쓰는 건 확인이 필요하다. 차트 데이터를 **조회**하는 것은 부작용이 없지만, **변경**하는 것은 사용자의 분석 환경을 바꾼다.

9개 도구를 이 기준으로 나눴다. 조회 2개는 즉시 실행, 변경 7개는 사용자 확인 후 실행. 데이터 탐색 시에는 끊김 없이 대화하다가, 차트를 실제로 수정할 때만 확인을 거친다.

## AI가 멈추고, 사용자가 만지고, AI가 이어간다

변경 도구가 실행되면 에이전트가 **일시 정지(interrupt)**한다. 에이전트의 전체 상태가 저장되고, 프론트엔드에 확인 UI가 나타난다. 사용자가 승인하면 프론트엔드가 차트 상태를 직접 수정하고, 그 결과를 에이전트에게 돌려보낸다. 에이전트는 정확히 멈춘 지점에서 이어간다.

핵심은 **interrupt가 프론트엔드에서 실제 차트 조작으로 이어진다**는 점이다. AI가 "지표를 추가하겠습니다"라고 말하고 멈추면, 사용자가 확인을 누른 순간 차트가 바뀌고, AI는 바뀐 차트를 기반으로 다음 분석을 이어간다. AI의 판단과 사용자의 조작이 하나의 흐름으로 엮인다.

여기서 하나 까다로운 문제가 있었다. AI가 "삼성전자와 SK하이닉스에 볼린저밴드 추가"를 처리할 때, 두 종목에 대해 도구를 **두 번 병렬로** 호출하면 interrupt가 두 개 동시에 발생한다. 프론트엔드는 하나의 interrupt만 처리할 수 있었기에 하나가 유실됐다.

해결은 **배치 처리**였다. 여러 종목을 한 번의 도구 호출에 배열로 받게 인터페이스를 설계하고, 시스템 프롬프트에서 "interrupt 도구는 반드시 한 번만 호출"을 명시하고, 미들웨어에서 병렬 호출을 런타임에 감지하는 세 겹의 방어를 적용했다. 프롬프트 지시만으로는 LLM이 무시할 수 있기 때문에, **코드로 강제하는 층**이 반드시 필요하다.

## 돌이켜보면

결국 만든 것은 **텍스트, 차트, 조작이 하나의 대화 안에서 흐르는 경험**이었다. AI가 말하면서 차트를 그리고, 사용자가 차트를 만지면 AI가 이어서 분석한다. 숫자를 나열하는 AI와 데이터를 보여주는 AI는 같은 모델이라도 사용자가 느끼는 가치가 완전히 다르다.
