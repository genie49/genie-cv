# 왜 이 스택인가 — Turborepo + Fastify + LangChain.js + GCP Pub/Sub

인플루언서 마케팅 캠페인을 자동화하려면 AI 에이전트가 핵심입니다. 에이전트 프레임워크를 먼저 정하고, 그에 맞춰 언어, 백엔드, 통신, 저장소를 순서대로 결정해 나간 과정을 정리합니다.

## LangGraph에서 시작하다

에이전트 프레임워크 선택의 시작점은 LangGraph였습니다. 캠페인 자동화는 단순한 질문-응답이 아니라, 기획 → 브리프 분석 → 크리에이터 매칭 → 가이드 생성으로 이어지는 복잡한 워크플로우입니다. 단계별로 분기하고, 실패하면 롤백하고, 사용자 확인이 필요하면 멈췄다가 다시 이어가야 합니다.

LangGraph는 이런 워크플로우를 상태 머신 기반 그래프로 구성할 수 있게 해줍니다. 노드 간 순환, 조건부 분기, 체크포인팅, Human-in-the-Loop 인터럽트까지 — 프로덕션 수준의 에이전트 워크플로우에 필요한 것들이 프레임워크 수준에서 지원됩니다.

LangGraph를 쓰기로 하면서 자연스럽게 LangChain 생태계 안에 들어오게 되었고, 이후 LangChain이 v1.0을 출시하면서 TypeScript 지원이 안정화되고, LangSmith 기반 Observability까지 제공하면서 이 생태계에 안주하게 되었습니다.

## 왜 Python이 아니라 TypeScript인가

LangChain은 Python 버전이 먼저 나왔고 커뮤니티도 더 큽니다. 그런데 우리 프론트엔드는 Next.js, 백엔드 API도 Node.js였습니다. 여기서 AI 서비스만 Python으로 분리하면 문제가 생깁니다.

캠페인 데이터 구조 하나가 바뀌면 Python 쪽과 TypeScript 쪽 모두 타입을 수정해야 합니다. 두 언어 간 타입 동기화는 수동으로 해야 하고, 누락되면 런타임에 터집니다.

TypeScript로 통일하면 타입을 한 곳에서 관리할 수 있습니다. 프론트엔드, API, 에이전트 서비스가 모두 같은 타입 패키지를 참조하므로, 캠페인 스키마가 바뀌면 한 곳만 수정하면 전체에 반영됩니다. Single Source of Truth. LangChain.js는 핵심 기능에서 Python 버전과 기능 패리티가 거의 동일했기 때문에 이 선택이 가능했습니다.

## NestJS에서 시작했다가 Fastify로

초기에는 NestJS로 백엔드를 구축했습니다. 그런데 LangChain.js를 붙이면서 문제가 발생했습니다.

NestJS는 CommonJS 기반인데, LangChain.js는 ESM(ECMAScript Modules) 전용입니다. 모듈 시스템이 충돌하면서 임포트 방식을 맞추느라 손이 많이 갔습니다. 단순히 설정을 바꾸는 수준이 아니라, 프레임워크와 라이브러리의 근본적인 모듈 시스템 불일치였습니다.

이 문제를 계기로 NestJS를 걷어내기로 했습니다. 대안으로 Hono와 Fastify를 검토했고, Bun 런타임도 고려했습니다. 결론은 Fastify + Node.js였습니다.

- **Hono vs Fastify**: Hono가 오버헤드는 더 적었지만, pnpm 모노레포 환경에서 프로세스가 한 번 올라간 뒤에는 유의미한 성능 차이가 없었습니다. Fastify가 생태계가 더 성숙하고 플러그인이 풍부했습니다.
- **Bun vs Node.js**: Bun의 속도가 매력적이었지만, pnpm 기반 모노레포 생태계와의 호환성을 우선했습니다.

## 앱은 가벼운 통로, 로직은 공유 패키지에

NestJS를 걷어내면서 설계 철학이 확립되었습니다. **핵심 비즈니스 로직과 타입은 공유 패키지에서 관리하고, 각 앱은 그 로직을 연결하는 가벼운 통로로만 동작한다.**

이 철학에 따라 7개 서비스의 무게가 달라졌습니다. HTTP 요청을 받아야 하는 4개 서비스(API, Auth, Chat, Tools)만 Fastify를 사용하고, 나머지 3개(Notification, Workflow, Workflow-History)는 HTTP 프레임워크 없이 Pub/Sub 메시지만 구독하는 순수 워커로 구성했습니다. 불필요한 오버헤드를 키우기보다 성능을 택한 것입니다.

이 접근이 Turborepo 모노레포와 맞물립니다. 공유 패키지에 타입, 메시징 계약, 데이터 접근 계층을 두고, 각 앱은 최소한의 연결 코드만 가집니다. 타입 하나가 바뀌면 모노레포 안에서 의존성 순서대로 빌드되고, 깨지는 곳은 한 번의 CI에서 즉시 확인됩니다.

## 서비스 간 통신 — 왜 메시지 큐가 필요했는가

7개 서비스가 서로 통신해야 하는데, 단순 HTTP 호출로는 불가능한 구조적 문제가 있었습니다.

AI 에이전트는 한 번의 응답을 생성하는 데 수십 초에서 몇 분이 걸립니다. LLM이 토큰 단위로 응답을 생성해 나가는 동안 중간중간 도구 호출도 추적하고 전달해야 합니다. 에이전트가 작업하는 동안 Chat 서비스는 토큰 청크, 도구 실행 시작/완료, TODO 업데이트 등을 실시간으로 받아 사용자에게 전달해야 합니다. HTTP 요청-응답으로는 이 흐름을 처리할 수 없습니다.

여기에 두 가지 요구사항이 더 있습니다. 하나의 워크플로우 실행 결과를 여러 Chat 인스턴스에 동시에 전달해야 하고(Fan-out), 스트리밍과 별개로 모든 이벤트를 History 서비스에 유실 없이 기록해야 합니다(No-loss). 이 패턴들은 메시지 큐 없이는 구현이 복잡해집니다.

그룹에서 이미 GCP를 메인 클라우드로 사용하고 있었고, 팀원 수가 매우 적었습니다. Kafka나 RabbitMQ 클러스터를 직접 운영할 여력이 없었기 때문에, 추후에 변경하더라도 일단 관리 포인트가 적은 온디맨드 서비스를 쓰자는 판단을 내렸습니다. GCP Pub/Sub는 토픽만 생성하면 프로비저닝, 스케일링, 모니터링이 자동이고, 메시지 실패 시 자동 재시도와 Dead Letter 토픽도 내장되어 있습니다.

## 3개 데이터 저장소

데이터 특성에 맞게 저장소를 분리했습니다.

- **PostgreSQL** — LangGraph 워크플로우 체크포인트 저장소. 에이전트 상태를 영속화하여 서버 재시작 후에도 워크플로우를 이어가야 하므로 트랜잭션 보장이 필수입니다. 추후 RAG 시스템 도입 가능성도 고려했습니다.
- **MongoDB** — 메인 데이터베이스. 캠페인, 브랜드, 사용자 등 비즈니스 데이터의 고성능 읽기/쓰기를 담당합니다.
- **Redis** — 워크플로우 실행 중 캠페인 스냅샷, 세션, 캐시 등 다양한 상태 관리 정책을 위해 사용합니다.

## 핵심 인사이트

- **프레임워크 선택이 언어를 결정하고, 언어가 나머지를 결정합니다.** LangGraph → TypeScript → Fastify → Turborepo 모노레포로 이어지는 선택은 하나의 흐름입니다.
- **앱은 가볍게, 로직은 공유 패키지에.** 쓸데없는 오버헤드를 키우기보다 가볍게 시작하고 필요할 때 쌓아 나가는 접근이 소규모 팀에 맞습니다.
- **관리 포인트를 줄이는 게 먼저.** 직접 운영할 여력이 없는 인프라는 클라우드 관리형 서비스에 맡기는 것이 현실적입니다.
