# EUC-KR 데이터베이스와의 사투

한국의 오래된 웹 서비스를 운영해본 사람이라면 EUC-KR이라는 단어에 한숨이 나올 것이다. VocaTokTok도 예외가 아니었다. 10년 넘게 운영된 MySQL에 수만 건의 영어 단어와 한글 뜻이 EUC-KR로 인코딩되어 있었다. Next.js + Prisma로 전면 리빌드하면서 이 인코딩 문제를 해결한 과정을 정리한다.

## DB 인코딩을 UTF-8로 전환한다

구 시스템은 PHP + MySQL 5.x 조합이었고, 테이블 캐릭터셋과 커넥션 charset 모두 `euckr`이었다. MySQL 8.x로 업그레이드하면서 `utf8mb4`로 전환했다.

| 항목 | Before | After |
|---|---|---|
| MySQL 버전 | 5.x | 8.x |
| 캐릭터셋 | euckr | utf8mb4 |
| 콜레이션 | euckr_korean_ci | utf8mb4_unicode_ci |

단순히 설정만 바꾸면 끝날 것 같지만, 실제로는 세 가지 함정이 있었다.

## 함정 1: 이중 인코딩

가장 흔한 문제다. EUC-KR 바이트가 UTF-8로 **다시 인코딩**된 데이터가 있었다. DB에는 "정상적인 UTF-8"처럼 보이지만, 실제로는 EUC-KR 바이트를 UTF-8로 한 번 더 감싼 것이다. 화면에 표시하면 깨진 한글이 나온다.

이중 인코딩된 데이터를 감지하려면 바이트 패턴을 분석해야 한다. 한글 UTF-8은 3바이트이고 첫 바이트가 특정 범위에 있다. 이 패턴에 맞지 않는 데이터를 찾아서 역변환했다.

## 함정 2: CP949와 EUC-KR의 혼재

EUC-KR과 CP949는 비슷하지만 다르다. CP949는 EUC-KR의 확장 인코딩으로, EUC-KR에 없는 한글(옛한글, 특수 글자)을 추가로 포함한다. 10년간 쌓인 데이터에 두 인코딩이 섞여 있었다.

EUC-KR로 디코딩하면 일부 글자가 깨지고, CP949로 디코딩하면 괜찮은 데이터가 있었다. 디코딩 실패 시 CP949로 폴백하는 체인을 만들었다.

## 함정 3: 런타임 파일 업로드

DB 전환은 한 번이면 끝나지만, **학원에서 올리는 파일은 계속 EUC-KR로 올라온다.** 선생님이 엑셀로 단어 목록을 만들어 업로드하는데, 한국의 엑셀은 CSV를 EUC-KR로 저장하는 경우가 많다.

파일 업로드 시 인코딩을 자동 감지하는 로직을 추가했다. 바이트 패턴으로 인코딩을 추정하고, UTF-8이 아니면 변환한다. 감지에 실패하면 EUC-KR → CP949 → UTF-8 순서로 폴백한다.

## 돌이켜보면

인코딩 문제의 핵심은 **"데이터가 한 종류의 인코딩으로만 되어 있지 않다"**는 것이었다. 10년간 쌓인 데이터에 EUC-KR, CP949, 이중 인코딩이 섞여 있고, 학원에서 올리는 새 파일도 EUC-KR이다. DB 전환은 일회성이지만, 런타임 인코딩 처리는 서비스가 운영되는 한 계속 필요하다.
