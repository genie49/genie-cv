# LangGraph 에이전트 설계

LangGraph를 활용한 RAG 에이전트 그래프 설계와 도구 바인딩 과정을 정리합니다.

## 왜 LangGraph인가?

단순 RAG 체인 대신 LangGraph를 사용한 이유는 에이전트가 도구 호출 여부를 스스로 판단하도록 하기 위해서입니다. 일반적인 인사말에는 RAG 검색 없이 바로 답변하고, 경력/프로젝트 관련 질문에만 RAG 도구를 호출합니다.

## 그래프 구조

```
START → Agent(Grok 4.1) → [도구 호출?] → RAG Search Tool → Agent → END
                          → [직접 답변?] → END
```

에이전트 노드가 LLM을 호출하고, 도구 호출이 필요하면 RAG 검색 도구를 실행한 후 다시 에이전트로 돌아옵니다.

## RAG Search Tool

LangChain Tool로 래핑된 RAG 검색 도구입니다. 사용자 질문을 받아 LanceDB에서 관련 문서를 검색하고 포맷팅된 결과를 반환합니다.

## 시스템 프롬프트

에이전트에게 김형진의 AI 어시스턴트 역할을 부여하고, 컨텍스트 기반 답변, 한국어 응답, 인용 규칙 등을 설정합니다.

## Grok 4.1 선택 이유

- 도구 호출(function calling) 지원이 안정적
- 한국어 답변 품질이 우수
- xAI API를 통해 접근 가능

## 인사이트

- LangGraph의 조건부 엣지로 도구 호출 여부를 자연스럽게 분기
- 시스템 프롬프트에 명확한 규칙을 설정하면 환각(hallucination) 감소
- 스트리밍 모드에서 토큰 단위 출력을 SSE로 전달
