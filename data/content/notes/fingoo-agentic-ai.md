# 세 번 갈아엎고 나서야 보였다 — AI에게 필요한 건 정교한 구조가 아니라 맥락이었다

처음에는 하나의 에이전트에 모든 도구를 넣었다. 그 다음에는 노드를 세밀하게 나눠서 그래프로 엮었다. 둘 다 기대만큼 동작하지 않았다. 세 번의 구조 변경을 거쳐 현재 형태에 도달한 과정을 정리한다.

## 도구를 전부 때려박았는데 잘 됐다

Vercel AI SDK 기반으로 빠르게 프로토타입을 만들었다. 주가 조회, 기술적 분석, 재무제표, 차트 생성, 포트폴리오 계산 — 도구를 하나의 에이전트에 전부 넣었다. 구조 같은 건 없었다. 그냥 도구를 다 줬다.

**놀랍게도 꽤 잘 동작했다.** "삼성전자 기술적 분석해줘"라고 하면 RSI 도구를 호출하고, 결과를 보고 MACD도 호출하고, 두 결과를 종합해서 답변을 만들었다. 도구 선택이 가끔 엉뚱하긴 했지만, 자기가 선택한 도구의 결과를 보고 다음 판단을 내리는 흐름이 작동했다. 뿌듯했다. 팀에게 "이걸 좀 더 세밀하게 엔지니어링하면 훨씬 좋아질 겁니다"라고 자신 있게 말했다.

하지만 한계도 보였다. 도구가 많다 보니 가끔 기술 분석을 요청했는데 포트폴리오 도구를 호출하거나, 같은 도구를 불필요하게 여러 번 호출하는 경우가 있었다. 대화가 길어지면 컨텍스트가 빠르게 차올라 후반부 응답 품질이 떨어졌다. "구조화하면 이런 문제가 사라지겠지"라고 생각했다.

## 세밀하게 짰더니 오히려 안 됐다

LangGraph로 전환해서 그래프 기반 에이전트를 설계했다. 질문 분류 노드, 데이터 수집 노드, 분석 노드, 응답 생성 노드 — 역할별로 노드를 나누고 라우팅을 세밀하게 짰다. "이 질문은 기술 분석이니까 이 노드로, 저 질문은 재무 분석이니까 저 노드로." 깔끔한 구조였다. 도구 선택 실수도 줄어들 거라고 확신했다.

결과는 **때려박은 것보다 못했다.**

도구 선택은 정확해졌지만, 분석의 깊이가 얕아졌다. "삼성전자 종합 분석해줘"라고 하면, 각 노드에서 자기 역할만 수행하고 넘겼다. 기술 분석 노드는 RSI만 계산하고, 재무 분석 노드는 PER만 조회하고 — 이전 노드의 결과를 참고해서 "그렇다면 이것도 확인해봐야겠다"는 **연쇄적 추론**이 사라졌다.

프롬프트를 바꿔봤다. 노드를 재배치했다. 라우팅 조건을 조정했다. 나아지지 않았다. "좀 더 다듬으면 좋아질 거야"라고 스스로에게 말했지만, 개발을 계속해도 성능이 올라가지 않았다. 초조했다. 팀에서 "그냥 이전 버전으로 하면 안 돼?"라는 말이 나왔을 때, 자신에 대한 실망이 컸다. 더 잘 만들겠다고 큰소리쳤는데, 구조 없이 때려박은 것보다 못한 결과를 내고 있었으니까.

## 내가 뭘 놓쳤을까

여기서 멈추고 생각했다. **왜 구조화된 게 비구조화보다 못한 건가.** 코드를 더 고치기 전에, 레퍼런스를 찾아보기로 했다.

LangChain 팀의 공식 문서와 블로그를 정독했다. Anthropic의 에이전트 설계 가이드를 읽었다. 그리고 결정적으로, LangChain이 Claude Code의 에이전트 구조를 분석한 [Deep Agent 리서치](https://blog.langchain.com/evaluating-deep-agents-our-learnings/)를 봤다. Claude Code가 코딩 벤치마크에서 높은 성능을 내는 구조를 분석한 내용이었다.

여기서 패턴이 보였다. 잘 동작하는 에이전트들에는 공통점이 있었다. 노드를 세밀하게 분기하는 게 아니라, **하나의 에이전트에게 충분한 도구와 맥락을 주고 자율적으로 판단하게 하는 것**이었다. 핵심은 에이전트가 이전 판단의 결과를 기억하고, 그걸 바탕으로 다음 행동을 결정하는 **연속적인 추론 루프**였다.

도구를 전부 때려박은 구조에서 에이전트가 어느 정도 동작한 이유가 여기에 있었다. 헷갈려 하더라도 자기 판단의 컨텍스트가 전부 남아 있었다. "아까 RSI를 봤더니 과매수 구간이었고, 그래서 MACD도 확인해봐야겠다고 판단했고, 결과가 이랬다." 이 맥락이 끊기지 않았다.

반면 노드를 세밀하게 나눈 구조에서는, 노드를 넘어갈 때마다 **컨텍스트가 깨졌다.** 이전 노드에서 왜 그런 판단을 내렸는지, 어떤 중간 결과를 보고 무엇을 느꼈는지가 다음 노드로 온전히 전달되지 않았다. 구조는 정교했지만, 에이전트 입장에서는 매 노드마다 기억을 잃는 셈이었다. 연쇄적 추론이 불가능한 건 당연했다.

**에이전트에게는 컨텍스트의 연속성이 정교한 라우팅보다 중요했다.**

## 큰 단위로 나누되, 안에서는 자유롭게

이 깨달음을 바탕으로 구조를 다시 설계했다. 노드를 세밀하게 나누는 대신, **영역별로 크게 나눈다.** Supervisor가 "이건 기술 분석 영역이야"라고 판단하면, 기술 분석 전문 에이전트에게 통째로 위임한다.

전문 에이전트 안에서는 자기 영역의 도구를 **자유롭게** 호출한다. RSI를 보고 "과매수 구간이네, 볼린저밴드도 확인해봐야겠다"는 연쇄적 추론이 살아난다. 자기 판단의 맥락이 끊기지 않으니까. 영역별로 전문 에이전트를 나눴다. 시장 데이터, 기술 분석, 리서치, 포트폴리오, 퀀트, 시각화, 지표 관리 — 각각 자기 영역의 도구만 가지고 있다. Supervisor는 도구를 직접 호출하지 않고 **"누구에게 무엇을 시킬지"만 판단**한다.

서브에이전트는 독립된 컨텍스트에서 실행되고, 끝나면 최종 결과만 Supervisor에 반환한다. 중간 과정은 폐기된다. 이렇게 하면 두 가지가 동시에 해결된다. Supervisor의 컨텍스트가 서브에이전트의 시행착오로 오염되지 않으면서, 각 서브에이전트 안에서는 자기 판단의 맥락이 온전히 유지된다.

1차(전부 한 곳에 — 맥락은 있지만 혼란)와 2차(너무 잘게 나눔 — 깔끔하지만 맥락이 깨짐) 사이의 **적정 지점**이었다. 도구 선택의 정확성과 추론의 연속성을 모두 가져갈 수 있는 구조.

## 미들웨어가 에이전트를 제어한다

구조가 잡히고 나니, 에이전트의 행동을 **안전하게 제어하는 층**이 필요해졌다. 에이전트에게 자율성을 주되, 일탈하면 안 되는 경계가 있다. 프롬프트로만 제어하면 LLM이 무시할 수 있다. 에이전트의 매 실행 단계를 가로채는 미들웨어 체인을 만들었다.

**모델 폴백** — 금융 서비스에서 "AI가 응답을 못합니다"는 치명적이다. 사용자가 시장이 급변하는 순간에 분석을 요청했는데 에러가 나면 신뢰를 잃는다. 주 모델이 장애를 일으키면 자동으로 대체 모델로 전환한다. 사용자는 응답 품질 차이만 느낄 뿐 에러를 보지 않는다.

**재귀 가드** — 에이전트가 도구 호출을 반복하다 무한 루프에 빠질 수 있다. 특히 서브에이전트가 병렬로 실행될 때, 각각이 도구를 여러 번 호출하면 전체 스텝 수가 급격히 늘어난다. 병렬 서브에이전트 수 × 평균 스텝 수를 계산해서 마진을 산정하고, 남은 스텝이 마진 이하면 도구를 제거하고 최종 응답을 강제한다.

**요약** — 금융 분석은 대화가 길어지기 쉽다. "삼성전자 분석해줘" → "하이닉스랑 비교해줘" → "반도체 섹터 전체는?" → "그중에 저평가된 종목은?" 한 세션에서 대화가 수십 턴 이어질 수 있다. 컨텍스트 윈도우 한계에 가까워지면, 이전 메시지를 LLM으로 요약해서 공간을 확보한다.

각 미들웨어는 하나의 관심사만 담당하고, 독립적으로 테스트할 수 있다. 에이전트 로직에 장애 처리, 모델 선택, 재귀 방지를 직접 섞으면 유지보수가 불가능해진다. 미들웨어로 분리하니 각각을 격리된 환경에서 검증할 수 있었다.

## 돌이켜보면

세 번의 구조 변경에서 배운 것은 결국 하나다. **에이전트의 성능은 구조의 정교함이 아니라 컨텍스트의 연속성에서 나온다.** 너무 풀어놓으면 도구 선택이 흐려지고, 너무 잘게 나누면 맥락이 깨진다. 영역별로 크게 나누되 안에서는 자유롭게 — 이 균형을 찾는 데 세 번이 걸렸다.

그리고 하나 더. 코드를 고치기 전에 레퍼런스를 찾아본 것이 전환점이었다. 혼자 삽질하며 "좀 더 다듬으면 나아지겠지"를 반복했다면 아마 네 번째, 다섯 번째 갈아엎기를 했을 것이다.
