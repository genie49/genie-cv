# 10년 된 코드를 새로 짓다

VocaTokTok은 10년 넘게 PHP + Vanilla JS로 운영된 영어 학습 플랫폼이다. 학원에서 매일 쓰는 서비스를 왜 전면 리빌드했는지, 그리고 Next.js + Prisma + MySQL 조합을 어떻게 구성했는지 정리한다.

## 유지보수의 한계

코드가 여기저기 흩어져 있었다. PHP 파일 안에 SQL과 HTML과 JavaScript가 뒤섞여 있고, 파일 간의 관계가 명확하지 않았다. 오류가 나면 어디서 발생한 건지 추적하는 것부터가 일이었다.

Vanilla JS 위에 jQuery로 뷰를 구성하고 관리하는 것도 쉬운 작업이 아니었다. 상태가 전역 변수에 흩어져 있고, DOM을 직접 조작하다 보니 한 곳을 고치면 다른 곳이 깨지는 일이 반복됐다. 새 기능을 추가할 때마다 기존 코드를 해독하는 데 더 많은 시간이 들었다.

## 왜 전면 리빌드인가

점진적 마이그레이션(Strangler Fig)이 일반적으로 안전하다고 알려져 있지만, VocaTokTok의 상황은 달랐다.

첫째, **재사용할 코드가 없다.** 비즈니스 로직이 뷰와 완전히 결합되어 있어서 추출 자체가 불가능하다. 점진적으로 전환해도 결국 모든 코드를 다시 작성해야 한다.

둘째, **핵심 자산은 코드가 아니라 DB다.** MySQL에 축적된 10년치 학습 데이터가 진짜 자산이다. 코드를 전부 갈아엎더라도 DB만 유지하면 된다.

셋째, **1인 개발 환경에서 병행 운영은 비현실적이다.** nginx에서 PHP와 Next.js를 경로별로 분기하면 세션 공유, 인증 동기화, CSS 충돌 등 부수 문제가 생긴다. 혼자서 두 시스템을 동시에 관리하면서 점진적으로 전환하는 건 현실적이지 않았다.

## Prisma — DB는 건드리지 않고 ORM만 교체한다

새 프레임워크를 도입하면서도 DB는 그대로 유지해야 했다. Prisma의 `db pull` 명령이 이 문제를 해결했다. 기존 MySQL 스키마를 introspect하여 **63개 모델**의 타입 안전한 스키마가 자동으로 만들어졌다. DB 마이그레이션 리스크 제로.

빌드 스크립트에 체이닝해서 빌드할 때마다 DB 스키마와 Prisma Client가 자동 동기화된다. MySQL을 PostgreSQL로 전환하지 않은 이유도 명확하다. 10년치 데이터가 있는 프로덕션 DB를 학원이 실시간으로 사용하고 있다. **ORM 레이어만 교체하는 것**이 가장 안전한 전략이었다.

## PWA — 학생들이 웹사이트를 찾아 들어오지 않는다

학원 학생들에게 "브라우저 열고 주소 치고 로그인해"는 너무 높은 허들이었다. 특히 스마트폰에서 매번 URL을 입력하는 걸 귀찮아했다. PWA로 **홈 화면에 앱 아이콘을 설치**할 수 있게 만들었다. 앱 스토어 없이도 앱처럼 접근할 수 있다.

## FTP에서 Vercel로 — 배포가 달라지다

PHP 시절의 배포는 FTP로 파일을 직접 올리는 방식이었다. 어떤 파일을 바꿨는지 기억해야 하고, 실수로 잘못된 파일을 올리면 즉시 장애가 났다.

Vercel은 Git과 연동되어 **main 브랜치에 푸시하면 자동 배포**된다. Preview 배포로 변경사항을 미리 확인하고, 문제가 있으면 이전 배포로 즉시 롤백한다. FTP 시절과는 비교할 수 없는 안정성이었다.

## Before → After

| 역할 | Before (PHP) | After (Next.js) |
|---|---|---|
| DB 접근 | mysqli raw SQL | Prisma ORM |
| 상태 관리 | 전역 변수 + jQuery | Zustand |
| 런타임 검증 | 없음 | Zod |
| 스타일링 | 인라인 CSS | Tailwind + Mantine |
| 모니터링 | 없음 | Sentry |
| 배포 | FTP 수동 | Vercel 자동 |
| 오프라인 | 불가 | PWA |

## 돌이켜보면

전면 리빌드가 정답인 경우가 있다. 재사용할 코드가 없고, 핵심 자산이 DB에 있고, 1인 개발이라 병행 운영이 비현실적이라면 — 새로 짓는 게 낫다. Prisma의 `db pull`이 이 전략을 가능하게 했다. 코드는 전부 바꾸되, 데이터는 한 바이트도 건드리지 않는다. 배포 파이프라인부터 모니터링까지, 기존에 없던 인프라가 갖춰지면서 유지보수의 한계를 느꼈던 서비스가 다시 발전할 수 있는 기반이 만들어졌다.
