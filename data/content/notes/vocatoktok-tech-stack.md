# 왜 이 스택인가 — Next.js 14 + Prisma + MySQL

VocaTokTok은 10년 넘게 PHP + Vanilla JS로 운영된 영어 학습 플랫폼이다. 새 기능을 추가할 때마다 jQuery 셀렉터를 추적하고, SQL이 뷰 파일에 섞인 코드를 해독해야 했다. 왜 점진적 마이그레이션이 아닌 전면 리빌드를 선택했는지, 새 스택을 어떻게 구성했는지 정리한다.

## 왜 전면 리빌드인가

점진적 마이그레이션(Strangler Fig)이 일반적으로 안전하다고 알려져 있지만, VocaTokTok의 상황은 달랐다.

첫째, **재사용할 코드가 없다.** 비즈니스 로직이 뷰와 완전히 결합되어 있어서 추출 자체가 불가능하다. 점진적으로 전환해도 결국 모든 코드를 다시 작성해야 한다.

둘째, **핵심 자산은 코드가 아니라 DB다.** MySQL에 축적된 10년치 학습 데이터가 진짜 자산이다. 코드를 전부 갈아엎더라도 DB만 유지하면 된다.

셋째, **1인 개발 환경에서 병행 운영은 비현실적이다.** nginx에서 PHP와 Next.js를 경로별로 분기하면 세션 공유, 인증 동기화, CSS 충돌 등 부수 문제가 생긴다.

## Next.js 14 — PHP 경험이 오히려 도왔다

여러 프레임워크를 검토했지만 Next.js 14의 App Router가 가장 적합했다.

PHP의 "서버에서 데이터 가져와서 렌더링"하는 패턴이 **Server Components에서 타입 안전하게 부활**한다. PHP 개발 경험이 오히려 자연스러운 전환을 도왔다. Route Groups로 학습 영역과 관리 영역을 깔끔하게 분리하고, API Route Handlers로 별도 백엔드 서버 없이 풀스택을 하나의 프로젝트에서 관리한다.

## Prisma — DB는 건드리지 않고 ORM만 교체한다

새 프레임워크를 도입하면서도 DB는 그대로 유지해야 했다. Prisma의 `db pull` 명령이 이 문제를 해결했다. 기존 MySQL 스키마를 introspect하여 **63개 모델**의 타입 안전한 스키마가 자동으로 만들어졌다. DB 마이그레이션 리스크 제로.

빌드 스크립트에 체이닝해서 빌드할 때마다 DB 스키마와 Prisma Client가 자동 동기화된다. MySQL을 PostgreSQL로 전환하지 않은 이유도 명확하다. 10년치 데이터가 있는 프로덕션 DB를 학원이 실시간으로 사용하고 있다. **ORM 레이어만 교체하는 것**이 가장 안전한 전략이었다.

## Before → After

| 역할 | Before (PHP) | After (Next.js 14) |
|---|---|---|
| DB 접근 | mysqli raw SQL | Prisma ORM |
| 상태 관리 | 전역 변수 + jQuery | Zustand |
| 런타임 검증 | 없음 | Zod |
| 스타일링 | 인라인 CSS | Tailwind + Mantine |
| 모니터링 | 없음 | Sentry |
| 배포 | FTP 수동 | Vercel 자동 |
| 오프라인 | 불가 | PWA |

## 돌이켜보면

전면 리빌드가 정답인 경우가 있다. 재사용할 코드가 없고, 핵심 자산이 DB에 있고, 1인 개발이라 병행 운영이 비현실적이라면 — 새로 짓는 게 낫다. Prisma의 `db pull`이 이 전략을 가능하게 했다. 코드는 전부 바꾸되, 데이터는 한 바이트도 건드리지 않는다.
