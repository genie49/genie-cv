# 이 에이전트 진짜 잘하고 있는 거 맞아?

프롬프트를 고치면 뭔가 좋아진 것 같다. 그런데 정말 좋아진 건지, 특정 케이스에서만 잘 되는 건지, 어제 고친 프롬프트가 다른 시나리오를 망가뜨린 건 아닌지 확신할 수 없다. LLM 출력은 비결정적이라 같은 입력에 다른 결과가 나온다. **"감"으로 튜닝하는 건 한계가 있었다.**

이건 우리만의 문제가 아니다. [LangChain의 2025 설문](https://www.langchain.com/state-of-agent-engineering)에 따르면 에이전트를 프로덕션에 운영하는 팀의 **52%만이 체계적 평가를 도입**했고, 품질 문제가 가장 큰 블로커(32%)로 꼽혔다. 절반은 여전히 감으로 판단하고 있다는 뜻이다.

[Anthropic의 에이전트 평가 블로그](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)에서 영감을 받았다. Task, Grader, Transcript — 이 세 가지 개념이 평가 시스템의 뼈대가 됐다.

## 블로그가 준 뼈대

블로그에서 정의한 용어가 명확했다.

| 개념 | 역할 |
|---|---|
| Task | 입력과 성공 기준이 정의된 하나의 테스트 시나리오 |
| Transcript | 에이전트 실행의 전체 기록 — 메시지, 도구 호출, 상태 변경 |
| Grader | Transcript를 채점하는 로직. 하나의 Task에 여러 Grader를 붙일 수 있다 |

그리고 하나의 원칙이 설계 전반을 관통했다. **"경로가 아니라 결과를 채점하라(Grade outcomes, not paths)."** 에이전트가 어떤 순서로 도구를 호출했는지보다, 최종 상태가 올바른지가 중요하다. 에이전트는 종종 설계자가 예상하지 못한 창의적인 경로를 찾아내기 때문이다.

이 뼈대 위에 실제 평가 시스템을 만들어 나갔다.

## 평가 환경을 만든다

에이전트를 평가하려면 먼저 **재현 가능한 환경**이 필요하다. 프로덕션 환경에서 평가를 돌리면 외부 API 상태, DB 데이터, 네트워크 지연 같은 변수가 결과에 영향을 준다. 같은 시나리오를 두 번 돌려도 다른 결과가 나올 수 있다 — LLM의 비결정성만으로도 충분히 복잡한데, 환경 변수까지 더하면 평가가 의미를 잃는다.

외부 의존성은 모킹하고, **LLM과 도구 호출 로직은 실제로 실행**한다. Redis 대신 인메모리 스냅샷 서비스를, 외부 API 대신 모킹 서비스를 넣었다. 평가하고 싶은 건 에이전트의 추론 능력이지 인프라의 안정성이 아니기 때문이다.

**데이터셋이 초기 조건을 고정**한다. "빈 캠페인에서 시작", "브리프까지 작성된 상태에서 시작", "전략까지 완료된 상태에서 시작" — 6개 데이터셋 템플릿이 캠페인의 다양한 단계를 재현한다. 같은 Task를 같은 데이터셋으로 돌리면, LLM의 비결정성을 제외한 모든 변수가 통제된다.

각 평가는 독립된 캠페인 ID로 실행되어 서로 간섭하지 않는다. **격리된 환경에서, 통제된 초기 조건으로, 실제 LLM을 돌린다.** 이게 평가의 기본 전제다.

## 코드로 잡을 수 있는 건 코드로

Grader를 설계할 때 원칙이 하나 있었다. **결정론적으로 판단할 수 있는 건 코드로 채점한다.** LLM 채점은 유연하지만 비용이 들고, 그 자체가 비결정적이다. 코드로 잡을 수 있으면 코드가 낫다.

4개의 코드 기반 Grader를 만들었다.

**도구 호출 검증(Tool Calls Grader)** — 에이전트가 올바른 도구를 올바른 파라미터로 호출했는지 확인한다. 4가지 매칭 모드가 있다. 정확히 이 도구들만 이 순서로(strict), 순서 무관하게 이 도구들만(unordered), 최소한 이 도구들은 포함(subset), 이 도구들을 초과하지 않음(superset).

**키워드 검증(Output Contains Grader)** — 응답에 특정 키워드가 포함되어 있는지 확인한다. AND 모드("A도 있고 B도 있어야"), OR 모드("A 또는 B가 있으면")를 지원한다.

**제약조건 검증(Constraints Grader)** — 턴 수, 도구 호출 횟수, 실행 시간의 상한/하한을 검증한다. "3턴 이내에 끝내야 한다", "도구를 5번 이상 호출하면 안 된다" 같은 효율성 기준.

**상태 검증(State Check Grader)** — 에이전트 실행 후 캠페인의 최종 상태를 검증한다. dot notation으로 `product.name`, `brief.budget` 같은 필드를 확인한다. 이게 "경로가 아니라 결과를 채점하라"는 원칙의 직접적인 구현이다. 에이전트가 어떤 순서로 도구를 호출했든, 최종 데이터가 올바르면 통과.

4개 Grader가 에이전트 실패의 대부분을 잡아낸다. 도구를 잘못 골랐거나, 필수 정보를 빠뜨렸거나, 비효율적으로 동작했거나, 데이터를 잘못 수정했거나.

## LLM이 LLM을 평가한다

코드로는 "응답이 좋은가?"를 판단할 수 없다. 정확성, 완전성, 전문성, 명확성 — 이런 건 사람이 읽어야 안다. 그런데 매번 사람이 읽을 수는 없다.

**루브릭 기반 LLM Grader**를 만들었다. 마크다운으로 작성된 채점 기준표(루브릭)를 LLM에게 주고, 에이전트의 응답을 0.0~1.0으로 채점하게 한다. 7개 루브릭이 있다 — 일반 응답 품질, 전문성 깊이, 대화 품질, 적응적 응답, 정보 추출 등 시나리오 유형별로 다른 기준을 적용한다.

핵심은 **루브릭의 구체성**이다. "좋은 응답인가?"라고 물으면 LLM도 모호하게 답한다. "정확성, 완전성, 명확성, 전문성 네 가지 기준으로, 각각 0.0~1.0으로 채점하고, 종합 0.7 이상이면 통과"라고 물으면 일관된 채점이 나온다.

## AI가 까다로운 사용자를 연기한다

싱글턴 평가만으로는 부족했다. 실제 사용자는 한 번 질문하고 끝나지 않는다. 여러 턴에 걸쳐 대화하고, 요구사항을 바꾸고, 모호한 질문을 던진다.

**롤플레이 평가**를 만들었다. LLM이 까다로운 사용자 역할을 연기하고, 에이전트와 멀티턴 대화를 나눈다. 6개 시나리오가 있다 — 전문적인 질문을 던지는 전문가 유저, 빠른 답변을 요구하는 급한 유저, 모호한 요청을 하는 유저 등.

흐름은 이렇다. 롤플레이어 LLM이 시나리오에 맞는 사용자 메시지를 생성하고, 에이전트가 응답하고, 판정자(Judge) LLM이 "목표에 얼마나 가까워졌는가"를 평가한다. 목표 달성, 계속 진행, 더 이상 진전 불가 — 세 가지 판정 중 하나를 내린다. 목표에 도달하거나 최대 턴에 도달할 때까지 반복한다.

싱글턴 평가가 **"이 질문에 잘 답하는가?"**라면, 롤플레이 평가는 **"이 대화를 잘 이끌어가는가?"**를 본다. 에이전트가 맥락을 유지하는지, 요구사항 변경에 적응하는지, 대화 전체에서 일관된 품질을 내는지.

## 평가 결과를 AI가 읽게 만든다

37개 Task, 6개 롤플레이 시나리오, 5개 Grader. 이 평가 시스템을 만든 진짜 이유는 **사람이 매번 읽으라고** 만든 게 아니다.

평가의 모든 결과 — 어떤 Task에서 어떤 Grader가 실패했고, Transcript에 어떤 도구 호출이 있었고, 루브릭 점수가 몇 점이었는지 — 가 **구조화된 형태로 저장**된다. 목표, 과정, 결과가 모두 기계가 읽을 수 있는 포맷이다.

이 구조가 코딩 에이전트(Claude Code)에게 열려 있다. 평가를 돌리고, 실패한 시나리오를 확인하고, Transcript를 분석해서 "이 시나리오에서 도구 선택이 잘못됐다 → 프롬프트의 이 부분을 수정하면 된다"는 추론을 AI가 직접 할 수 있다. 사람이 실패 원인을 분석하고 프롬프트를 고치는 대신, **AI가 평가 결과를 읽고 스스로 개선 방향을 찾는 구조**다.

평가 시스템의 가치는 점수 자체가 아니다. **개선 루프의 속도**다. 평가 → 분석 → 수정 → 재평가, 이 사이클이 빨라질수록 에이전트 품질이 빠르게 올라간다. 사람이 병목이 되지 않는 구조를 만든 것이 이 평가 시스템의 핵심이었다.

## 돌이켜보면

37개 Task는 어디서 왔을까. 대부분 **프로덕션에서 발견한 실패**다. 도구를 잘못 고른 케이스, 엉뚱한 데이터를 수정한 케이스, 사용자 의도를 오해한 케이스 — 실패할 때마다 그 시나리오를 Task로 변환했다. [OpenAI가 설명하는 Eval Flywheel](https://cookbook.openai.com/examples/evaluation/building_resilient_prompts_using_an_evaluation_flywheel)과 같은 구조다. 배포 → 실패 발견 → 테스트 케이스 추가 → 개선 → 재배포. 이 바퀴가 돌수록 평가 데이터셋이 두꺼워지고, 에이전트가 같은 실수를 반복할 확률이 줄어든다.

결국 평가 시스템은 한 번 만들고 끝나는 게 아니라 **프로덕션과 함께 성장하는 구조**다. 에이전트가 새로운 방식으로 실패할 때마다 평가가 더 정밀해지고, 평가가 정밀해질수록 다음 개선의 방향이 명확해진다.
